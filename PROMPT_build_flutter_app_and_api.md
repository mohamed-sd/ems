## الهدف العام

ابنِ نظاماً متكاملاً من جزأين على نظام EMS الحالي (PHP + MySQL، عربي RTL، متعدّد الشركات):

1. **طبقة REST API** (PHP) تعمل على نفس قاعدة بيانات النظام وتغطّي عمليات شاشتي مدير الحركة والتشغيل.
2. **تطبيق Flutter كامل وجاهز** يستهلك هذه الـ API، مطابق لملف التصميم المرفق، يغطّي شاشتين فقط + الدخول.

التطبيق مخصّص لدور **مدير الحركة والتشغيل**، والمشروع الذي يعمل عليه **يُعرف تلقائياً من حسابه عند الدخول** (لكل مستخدم `project_id` مخزّن في جدول `users` ويُحمّل في الجلسة). لا توجد شاشة اختيار مشروع.

## المصادر التي يجب قراءتها أولاً (مصدر الحقيقة لمنطق العمل)

اقرأ هذه الملفات بالكامل قبل أي كتابة، فالـ API يجب أن **تطابق منطقها وقواعدها واستعلاماتها** بدقّة:

- `movement/map_page.php` — منطق وبيانات **اللوحة الحيّة/الخريطة**.
- `movement/movement_operations.php` — منطق وعمليات **الحركة والتشغيل الموحّدة** (add_new_operation, save_single_operation, add_new_driver, save_single_driver).
- `config.php` — اتصال قاعدة البيانات والإعدادات.
- `includes/permissions_helper.php` — دوال الصلاحيات (`check_page_permissions`) وعزل الشركة.
- `login.php` — كيف تُبنى جلسة المستخدم (role, company_id, project_id).
- `BRIEF_flutter_movement_manager.md` — الوصف الوظيفي الكامل للشاشتين والقواعد.
- ملف التصميم المرفق (المسار أعلاه) — المرجع البصري الملزِم للتطبيق.

# الجزء الأول: طبقة الـ API (PHP)

## المبادئ

- ضعها في مجلد جديد **`api/`** داخل المشروع، **دون كسر** أي شيء في النظام الحالي.
- أعد استخدام اتصال قاعدة البيانات من `config.php` ودوال `db_table_has_column` والصلاحيات وعزل الشركة كما هي.
- **استبدل مصادقة الجلسة/الكوكيز + CSRF بمصادقة Token (Bearer)** مناسبة للجوال: عند الدخول يُصدَر توكن يُخزَّن في جدول جديد `api_tokens` (id, user_id, token_hash, created_at, expires_at, revoked)، ويُرسل في كل طلب عبر `Authorization: Bearer <token>`. لا تستخدم CSRF في الـ API.
- **كل الاستعلامات Prepared Statements**، وكل الردود JSON موحّدة: `{ "success": bool, "message": string, "data": mixed }` مع `JSON_UNESCAPED_UNICODE` وترويسة `Content-Type: application/json; charset=utf-8`.
- **عزل صارم:** كل endpoint يتحقق من التوكن ويستخرج `user_id, role, company_id, project_id`، ويقصر كل البيانات على شركة المستخدم ومشروعه (إلا السوبر أدمن role=-1).
- رسائل الأخطاء بالعربية وبنفس أسلوب النظام، مع أكواد HTTP صحيحة (401 غير مصرّح، 403 لا صلاحية، 400 إدخال خاطئ، 404 غير موجود، 422 فشل تحقّق، 500 خطأ خادم).
- نقطة معالجة أخطاء مركزية (مثل `excel.php`) تحوّل أي استثناء/خطأ فادح إلى JSON واضح وتسجّله.

## نقاط النهاية المطلوبة (Endpoints)

**المصادقة والسياق:**
- `POST /api/login` — مدخلات: username, password. يتحقق (bcrypt كما في login.php)، ويُصدر توكناً، ويعيد: التوكن + بيانات المستخدم (id, name, role, company_id, project_id) + **بيانات المشروع** (id, name, project_code, location, الإحداثيات).
- `POST /api/logout` — إبطال التوكن الحالي.
- `GET /api/me` — بيانات المستخدم والمشروع الحالي (للاستئناف بعد فتح التطبيق).

**اللوحة الحيّة (مكافئ map_page):**
- `GET /api/board` — يعيد كل بيانات اللوحة لمشروع المستخدم:
  - `stats`: { suppliers, equipment_total, equipment_working, equipment_stopped, operators }.
  - `suppliers[]`: لكل مورّد { supplier_id, supplier_name, totals{...}, equipments[] }.
  - كل `equipment`: { op_id, eq_id, code, name, type_name, is_working, availability_status, manufacturer, model, serial_number, chassis_number, ts_total, ts_today, drivers[] }.
  - كل `driver`: { driver_id, name, driver_code, phone, skill_level, license_type, years_in_field, years_on_equipment }.
  - استخرج نفس منطق التجميع حسب المورّد وحساب الساعات من التايم شيت كما في `map_page.php` تماماً.

**الحركة والتشغيل الموحّدة (مكافئ movement_operations):**
- `GET /api/operations` — عمليات المشروع مقسّمة جاهزة: قائمة المعدات مع { بيانات التشغيل، الفئة، الوردية، الحالة، البداية/النهاية، عدّاد السائقين، drivers[] (تعيينات كل معدة)، إحداثيات للخريطة }. أرفِق ما يلزم لبناء جدولي النهار (D,B) والليل (N,B) ودبابيس الخريطة.
- `POST /api/operations` — إضافة تشغيل (add_new_operation): equipment, equipment_type, contract_id, supplier_id, equipment_category, shift_type, start, end, total_equipment_hours, shift_hours, status. **يمنع** إضافة معدة لها تشغيل ساري آخر.
- `PUT /api/operations/{op_id}` — تعديل/إنهاء تشغيل (save_single_operation): equipment_category, shift_type, status(1/0), start, end. (status=0 = إنهاء.) تحقّق صيغة التواريخ والنهاية بعد البداية، ومقيّد بالمشروع والشركة.
- `POST /api/equipment-drivers` — إضافة سائق لمعدة (add_new_driver): driver_id, equipment_id, shift_type, start_date, end_date. يتحقق أن المعدة مشغّلة في تشغيل ساري، و**يُنهي تلقائياً** تعيينات هذا السائق السارية السابقة.
- `PUT /api/equipment-drivers/{rel_id}` — تعديل/إنهاء تعيين سائق (save_single_driver): shift_type, start_date, end_date, status. (end_date فارغ → 2099-12-31.)

**قوائم مساعدة (لنماذج الإضافة):**
- `GET /api/contracts` — عقود المشروع النشطة.
- `GET /api/suppliers` — موردو المشروع.
- `GET /api/equipment-types` — أنواع المعدات النشطة.
- `GET /api/equipments?type=&supplier=` — المعدات المتاحة حسب النوع/المورّد.
- `GET /api/drivers/available?equipment_id=` — السائقون المتاحون (نشطون، ضمن المشروع/الشركة، مستثنى أصحاب التعيين الساري).

> طابِق **القواعد الصارمة** الموجودة في الشاشات: لا تشغيل مزدوج لمعدة، سائق واحد لا يكون نشطاً على أكثر من تعيين، التحرير للسجلات السارية فقط، وقيم الوردية D/N/B والفئة (أساسي/احتياطي).

## توثيق الـ API

أنشئ `api/README.md` يوثّق كل endpoint: المسار، الطريقة، المدخلات، المخرجات، أكواد الأخطاء، ومثال طلب/استجابة. هذا الملف سيكون مرجع تطبيق Flutter.

# الجزء الثاني: تطبيق Flutter

## المتطلبات

- **نفّذ ملف التصميم المرفق بدقّة** (الشاشات، الألوان، الخطوط، المكوّنات، التدفّقات). إن نقص في التصميم تفصيل وظيفي، عُد إلى `BRIEF_flutter_movement_manager.md`.
- مشروع Flutter كامل وجاهز للتشغيل في مجلد جديد منفصل (مثل `flutter_app/` داخل المستودع أو حسب تفضيلك — وضّحه).
- **عربي بالكامل، RTL** (`Directionality` / `MaterialApp` مع `locale` عربي و`supportedLocales`).
- **الشاشات:**
  1. **الدخول** — username/password → استدعاء `/api/login` → حفظ التوكن بأمان (`flutter_secure_storage`) → الدخول مباشرة على مشروع المستخدم (بلا اختيار مشروع). اعرض اسم المشروع في الرأس.
  2. **اللوحة الحيّة/الخريطة** — من `/api/board`: البطاقات الإحصائية الخمسة، مفتاح الألوان، تجميع الموردين، بطاقات الآليات (عاملة/متوقفة)، تفاصيل الآلية والمشغّل (Bottom sheet/Dialog)، خريطة بالدبابيس، شارة «عرض مباشر»، **سحب للتحديث**.
  3. **الحركة والتشغيل الموحّدة** — من `/api/operations`: جدولا/قسما النهار والليل، صف فرعي قابل للتوسيع لسائقي كل معدة، نماذج تعديل التشغيل والسائق مع **حفظ فوري لكل صف** (استدعاءات PUT/POST)، نموذج إضافة سائق، والخريطة. أخفِ/أظهر أزرار التعديل حسب الصلاحية، واجعل التحرير للسجلات السارية فقط.

## البنية والجودة

- معمارية نظيفة وطبقات واضحة: طبقة API client (Dio أو http) + models + repositories + إدارة حالة (Provider/Riverpod/Bloc — اختر وبرّر) + شاشات/widgets.
- **عميل API مركزي:** يضيف `Authorization: Bearer` تلقائياً، يعالج 401 (إخراج/إعادة دخول)، ويوحّد قراءة `{success, message, data}` وعرض الأخطاء العربية كـ Snackbar/Toast.
- **مكوّنات قابلة لإعادة الاستخدام:** بطاقة آلية، بطاقة مورّد، شارات الوردية (☀️/🌙/🔄) والحالة (أخضر/أحمر/رمادي)، صف قابل للتوسّع، قوائم متسلسلة (العقد→المورّد→النوع→المعدة)، حقول تاريخ.
- **معالجة الاصطلاحات:** نهاية التعيين `2099-12-31` تُعرض «مستمر»/فارغة؛ التواريخ بصيغة `YYYY-MM-DD`؛ ألوان دلالية متّسقة.
- حالات تحميل/فراغ/خطأ أنيقة لكل شاشة.
- وفّر `flutter_app/README.md`: كيف تُشغّل، أين يُضبط `base_url` للـ API، التبعيات.

# التحقق (إلزامي قبل الإنهاء)

**للـ API:**
1. `php -l` على كل ملفات `api/` — بلا أخطاء.
2. اختبر يدوياً (curl/Postman) المسارات الأساسية: login → board → operations → POST/PUT operation → POST/PUT equipment-driver، وتأكّد من: صحّة الردود، تطبيق العزل، تطبيق القواعد الصارمة (جرّب تشغيلاً مزدوجاً لمعدة وتأكّد من الرفض)، وأكواد الأخطاء.
3. تأكّد أن النظام الويب الحالي لم يتأثّر (الـ API معزولة في `api/`).

**للتطبيق:**
4. `flutter analyze` بلا أخطاء، و`flutter build` ينجح.
5. شغّل التطبيق على محاكي وتتبّع التدفّق الكامل: دخول → ظهور المشروع تلقائياً → اللوحة الحيّة بكل بياناتها → الحركة والتشغيل (تعديل تشغيل، إضافة/تعديل سائق، حفظ فوري) → التأكد من سلامة RTL والعربية والألوان والخريطة.
