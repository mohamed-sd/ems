# برومت تنفيذ: فصل «دور الآلية» عن «حالتها التشغيلية» في الحركة والتشغيل

**التاريخ:** 2026-06-23
**النطاق:** `operations` (قاعدة بيانات) · `movement/movement_operations.php` · `Oprators/oprators.php`
**الهدف:** إعادة هيكلة منطق حالات الآليات بحيث يُفصَل **الدور** (أساسي/احتياطي) عن **الحالة التشغيلية**
(تعمل/جاهزة/معطلة) عن **دورة الحياة** (نشط/منتهي)، وحسم مصدر الحقيقة لكل محور.

---

## 0) الخلفية والمشكلة الجذرية

اليوم يحمل عمود واحد `operations.equipment_category` قيمًا تخلط **محورين مختلفين**:
الدور (أساسي/احتياطي) **و** حالة العطل (متعطل). والنتيجة أن زر «صيانة» في صفحة الحركة
(`set_equipment_maintenance`) **يدهس الدور** بتحويله إلى «متعطل»، فاضطُرّ النظام لعمود التفافي
`prev_equipment_category` لمحاولة استعادته لاحقًا. هذا هشّ ويُفقد التمييز أساسي/احتياطي اللازم للتقارير.

**الحل:** ثلاثة محاور مستقلة، وكل صفحة تكتب في محورها فقط.

---

## 1) القواعد الحاكمة (القرارات المعتمدة — لا تُغيّرها)

1. **محور الدور** `equipment_category` ∈ {`أساسي`,`احتياطي`} — يُحدَّد **مرة واحدة عند الإضافة من
   `Oprators/oprators.php` فقط**، وثابت لا يُبدَّل أبدًا. صفحة الحركة **لا تكتب فيه إطلاقًا**.
   (تُستخدم القيمة لاحقًا في التقارير وفي تقسيم جدولَي «أساسية/احتياطية» بصفحة التشغيل.)
2. **محور الحالة التشغيلية** `op_state` ∈ {`تعمل`,`جاهزة`,`معطلة`} — عمود **جديد**، يُدار
   **حصريًا من `movement/movement_operations.php`**. ينطبق على التشغيلات النشطة فقط (`status=1`).
3. **محور دورة الحياة** `status` ∈ {1 نشط, 0 منتهي} — يبقى كما هو. المنتهي = **history** بلا أي حالة.
4. **الحالة الابتدائية:** كل معدة تُضاف من مدير الأسطول تبدأ **`جاهزة`** (القيمة الافتراضية للعمود)،
   حتى يحوّلها مدير الحركة يدويًا إلى `تعمل`.
5. **منطق التبديل:** **تغيير منفرد يدوي** — لا تبديل مقترن ولا فرض. مدير الحركة يغيّر حالة كل صف على حدة
   (يُسمح منطقيًا بوجود أكثر من «تعمل»، فالتحكّم بشري).
6. **العودة من الصيانة:** بعد إغلاق أمر الصيانة، المعدة تعود إلى **`جاهزة`** (وليس `تعمل`) حتى يشغّلها المسؤول.
7. **إلغاء `prev_equipment_category`:** يصبح مهجورًا؛ أوقف الكتابة إليه (لا داعي لحذف العمود فورًا).

## 2) آلة الحالات (الانتقالات — كلها أفعال صريحة من المستخدم)

| الانتقال | المُحفِّز | الأثر |
|---------|----------|-------|
| (إنشاء) → `جاهزة` | إضافة معدة من صفحة التشغيل | `op_state='جاهزة'` افتراضيًا |
| `جاهزة` → `تعمل` | زر/محدّد في صفحة الحركة | `op_state='تعمل'` فقط (بلا اقتران) |
| `تعمل` → `جاهزة` | زر/محدّد في صفحة الحركة | `op_state='جاهزة'` فقط |
| `تعمل`/`جاهزة` → `معطلة` | زر «صيانة» | `op_state='معطلة'` + `equipments.availability_status='تحت الصيانة'` |
| `معطلة` → `جاهزة` | إغلاق أمر الصيانة / زر «إرجاع من الصيانة» | `op_state='جاهزة'` + `equipments.availability_status='قيد الاستخدام'` |
| نشط → منتهي | زر «إنهاء» | `status=0` (history، تُهمَل الحالة) |

> تكامل: انتقال «معطلة» هو نقطة الالتقاء مع ميزة **أمر الصيانة التلقائي** الموصوفة في
> `PROMPT_auto_maintenance_order.md`. نسّق التنفيذ بينهما (نفس بلوك `set_equipment_maintenance`).

## 3) قاعدة البيانات

### 3.1 العمود الجديد
`database/migrations/2026_06_23_operations_op_state.sql`
```sql
-- محور الحالة التشغيلية المستقل عن الدور
ALTER TABLE operations
  ADD COLUMN op_state ENUM('تعمل','جاهزة','معطلة') NOT NULL DEFAULT 'جاهزة'
  COMMENT 'الحالة التشغيلية للآلية النشطة (تُدار من صفحة الحركة فقط)'
  AFTER status;
```

### 3.2 ترحيل البيانات القائمة (تفكيك المحور المختلط)
```sql
-- 1) الصفوف «المتعطلة» سابقًا: انقل العطل إلى op_state، واسترجع الدور الحقيقي من العمود الالتفافي
UPDATE operations
SET op_state = 'معطلة',
    equipment_category = COALESCE(NULLIF(prev_equipment_category, ''), 'أساسي')
WHERE equipment_category = 'متعطل';

-- 2) باقي التشغيلات النشطة غير المعطلة: ابقها على الافتراضي «جاهزة»
--    (مدير الحركة يرفع ما يعمل فعليًا إلى «تعمل»). عدّل إن رغبت ببدء النشطة الحالية بـ «تعمل».
UPDATE operations
SET op_state = 'جاهزة'
WHERE status = 1 AND equipment_category <> 'متعطل';

-- 3) (اختياري للاتساق) المنتهية لا حالة لها فعليًا؛ اتركها على الافتراضي.
```
> طبّق المايجريشن يدويًا على قاعدة `localhost` قبل الاختبار. **خذ نسخة احتياطية أولًا.**

### 3.3 مصدر الحقيقة
- `operations.op_state` = مصدر الحقيقة للحالة التشغيلية داخل المشروع.
- `equipments.availability_status` = **مرآة عامة** للمعدة، تتزامن فقط عند الدخول/الخروج من «معطلة»
  (الجسر مع وحدة الصيانة). لا تُشتق حالة الصفحة منها بعد الآن.

## 4) `movement/movement_operations.php`

### 4.1 إيقاف الكتابة في عمود الدور
- في `set_equipment_maintenance` (حوالى السطر 432): **احذف** كتلتَي تحديث `equipment_category='متعطل'`
  وحفظ `prev_equipment_category` (حوالى الأسطر 469–485). استبدلها بتحديث `op_state='معطلة'`:
  ```php
  mysqli_query($conn, "UPDATE operations
                          SET op_state = 'معطلة'
                        WHERE equipment = $equipment_id
                          AND project_id = $selected_project_id
                          AND status = 1
                          $operations_company_scope_inline");
  ```
  (يبقى تحديث `equipments.availability_status='تحت الصيانة'` كما هو.)
- في `save_single_operation` (حوالى السطر 133): **أزل** أي كتابة لـ `equipment_category` من هذه الشاشة
  (التعليق في الكود يشير أصلًا إلى عدم تعديله) — صفحة الحركة لم تعد تمسّ الدور.

### 4.2 إجراءات جديدة (POST handlers)
أضِف معالجات جديدة ضمن نفس بنية `try/catch` و`json`:
1. `set_op_state` — لتبديل `تعمل ⇄ جاهزة` يدويًا:
   ```php
   $new_state = in_array($_POST['op_state'] ?? '', ['تعمل','جاهزة'], true) ? $_POST['op_state'] : null;
   // امنع ضبط «معطلة» من هنا (تُضبط عبر زر الصيانة فقط)
   // UPDATE operations SET op_state = ? WHERE id = op_id AND project_id ... AND status = 1
   ```
2. `restore_from_maintenance` — لانتقال `معطلة → جاهزة`:
   ```php
   // UPDATE operations SET op_state = 'جاهزة' WHERE equipment = ? AND project_id ... AND status = 1
   // UPDATE equipments SET availability_status = 'قيد الاستخدام', availability_state='متوفرة' WHERE id = ?
   ```
   استخدم prepared statements والتزم بعزل الشركة `$operations_company_scope_inline` / `$company_id`.

### 4.3 الاستعلام
في `$operations_sql` (حوالى السطر 511): أضِف `o.op_state` إلى الأعمدة المجلوبة.

### 4.4 طبقة العرض (حوالى الأسطر 1205–1279)
- **احذف** منطق `$is_under_maint` المشتق من `availability_status` (السطر 1211–1212) والشارة المزدوجة.
- للصفوف **النشطة**: اعرض شارة واحدة من `op_state` بثلاثة ألوان:
  - `تعمل` → أخضر، `جاهزة` → كهرماني/أزرق، `معطلة` → أحمر.
- للصفوف **المنتهية**: لا شارة حالة — تواريخ فقط (history).
- **الأزرار** (للصف النشط و`$can_edit`):
  - محدّد/زرّان لتبديل `تعمل ⇄ جاهزة` (يستدعي `set_op_state`).
  - زر «صيانة» يظهر إن لم تكن `op_state='معطلة'` (يستدعي الحالي → معطلة).
  - زر «إرجاع من الصيانة» يظهر فقط إن `op_state='معطلة'` (يستدعي `restore_from_maintenance`).
  - زر «إنهاء» كما هو.
- أضِف فئات CSS للشارات الثلاث (على نمط `.status-running`/`.status-maint` القائمة).

## 6) خطوات الاختبار (Verification — إلزامية)

1. **المايجريشن والترحيل:** بعد التطبيق، تحقّق أن كل صف كان `equipment_category='متعطل'` صار
   `op_state='معطلة'` ودوره استُرجع (أساسي/احتياطي) صحيحًا، ولا صف دوره = «متعطل».
2. **الإضافة:** أضِف معدة من صفحة التشغيل ⇒ تظهر في صفحة الحركة بحالة **«جاهزة»**.
3. **التبديل اليدوي:** جاهزة → تعمل ثم تعمل → جاهزة، كلٌّ على حدة، دون أي تأثير على صف آخر.
4. **الصيانة:** تعمل/جاهزة → معطلة ⇒ `op_state='معطلة'` + `availability_status='تحت الصيانة'`،
   والدور (أساسي/احتياطي) **لم يتغيّر**.
5. **العودة:** «إرجاع من الصيانة» (أو إغلاق أمر الصيانة) ⇒ المعدة تعود **«جاهزة»** و`availability_status='قيد الاستخدام'`.
6. **التقارير/التقسيم:** معدة أساسية مرّت بكل الحالات تبقى في جدول «الأساسية» بصفحة التشغيل طوال الوقت.
7. **المنتهية:** التشغيل المنتهي (`status=0`) لا يعرض أي شارة حالة.
8. **العزل والأمان:** كل التحديثات ضمن نطاق `company_id`، عبر prepared statements، والإخراج عبر `htmlspecialchars(ENT_QUOTES)`، ولا mojibake في النصوص العربية.
