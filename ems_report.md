# القسم الأول: الملخص التنفيذي

## 1.1 وصف النظام

النظام عبارة عن منصة **SaaS متعددة المستأجرين (Multi-Tenant)** متخصصة في **إدارة المعدات الثقيلة والأساطيل والمناجم والعقود** لقطاع التعدين والإنشاءات في السودان. يُسوّق تحت اسم **«منصة إنجاز»** / **Equipation**.

**الوحدات الرئيسية المكتشفة:**

| الوحدة | الوظيفة |
|--------|---------|
| المعدات (Equipments) | تسجيل المعدات، الأنواع، الأسطول، السائقين على المعدة، أعطال المعدات، ملف المعدة |
| السائقون (Drivers) | السائقون، عقود السائقين، سجل المركبات، ملف السائق |
| المشغّلون (Operators) | المشغّلون وربطهم بالعقود والمناجم |
| الموردون (Suppliers) | الموردون، عقود الموردين، معدات عقد المورد |
| العملاء (Clients) | العملاء وملفاتهم |
| المشاريع (Projects) | المشاريع، مناجم المشروع، مشاريع التشغيل |
| العقود (Contracts) | عقود رئيسية + عقود موردين + عقود سائقين (3 مسارات منفصلة) |
| كشوف الساعات (Timesheet) | إدخال ساعات التشغيل، أنواع الكشوف، حقول الحفر والنقل، الأعطال |
| الحركة والتشغيل (Movement) | تحريك السائقين/المشغّلين، عمليات التشغيل، الورديات، خريطة المواقع |
| الموافقات (Approvals) | طلبات الموافقة، موافقة الساعات، قواعد سير العمل |
| التقارير (Reports + emsreports) | 13 + 27 تقريراً (تفصيلي/ملخص) |
| سجلات النشاط (ActivityLogs) | تتبّع نشاط المستخدمين عبر بنية Job/Queue/Repository |
| الإعدادات (Settings) | الأدوار، صلاحيات الأدوار، الوحدات، تغيير كلمة المرور |
| لوحة الإدارة الفائقة (admin) | الشركات، الخطط، الاشتراكات، المدراء، الدعم، سجل التدقيق |

**المنظومة التقنية (Tech Stack):**
- **اللغة:** PHP إجرائي (Procedural) بالاعتماد على `mysqli`.
- **قاعدة البيانات:** MySQL / MariaDB (محرك InnoDB، 35 جدولاً، 60 مفتاحاً أجنبياً، 152 فهرساً/مفتاحاً).
- **الواجهة:** HTML/CSS/JS مكتوبة يدوياً (لا إطار أمامي)، خط RTL عربي.
- **الاعتماديات (Composer):** `phpoffice/phpspreadsheet` فقط (لتصدير/استيراد Excel).
- **بنية حديثة جزئية:** مجلد `app/` فيه `Services / Repositories / Jobs / Queues / Middleware / Observers` — لكنها **مستخدمة فقط لسجلات النشاط**، وبقية النظام إجرائي بالكامل.
- **البيئة:** WAMP على Windows؛ والإنتاج على استضافة Hostinger (وفق التعليقات في `config.php`).

## 1.3 أهم نقاط القوة

1. **نواة أمنية مركزية ناضجة** (`includes/security.php`): جلسات صارمة (`SameSite=Strict`, `HttpOnly`, تجديد دوري، بصمة الجلسة، مهلة)، CSRF، رؤوس أمنية + CSP، تحقق رفع الملفات، Rate Limiting.
2. **تسجيل الدخول آمن:** Prepared Statement + `password_verify` (bcrypt) + قفل بعد 5 محاولات + قيود طول.
3. **عزل المستأجرين (Multi-Tenancy)** مُطبّق فعلياً عبر `company_id` ومنطق نطاق على مستوى الاستعلام.
4. **قاعدة بيانات منظّمة نسبياً:** 60 مفتاحاً أجنبياً و152 فهرساً (ليست قاعدة "مسطّحة").
5. **بنية تسجيل نشاط حديثة** (Job/Queue/Repository) قابلة للتعميم على باقي النظام.
6. **تغطية وظيفية واسعة** للعمليات الميدانية (عقود متعددة الأنواع، كشوف ساعات تفصيلية، موافقات).
7. **معالجة الترميز العربي** مأخوذة بجدية (توثيق ومعايير ترميز).

## 1.4 أهم نقاط الضعف

1. **تنفيذ DDL (تعديل بنية الجداول) أثناء معالجة الطلب** في 13+ ملفاً — أخطر مشكلة معمارية.
2. **كلمات مرور قاعدة بيانات الإنتاج مكتوبة كنص صريح** داخل `config.php` (ومحفوظة في تاريخ Git).
3. **ملفات عملاقة أحادية** (حتى 2623 سطراً) تخلط كل الطبقات.
4. **بناء SQL بالتسلسل النصّي** (766 استعلاماً خاماً مقابل 139 Prepared).
5. **ثغرات تفويض على مستوى الإجراء** في معالجات AJAX (الحماية بالجلسة + XHR + Rate Limit فقط، دون فحص صلاحية الوحدة في كثير من الحالات).
6. **غياب كامل للاختبارات الآلية** (يوجد فقط مقارنة لقطات شاشة).
7. **أرقام سحرية للأدوار** (`role == "10"`, `"-1"`, `"3"`...) منتشرة في كل النظام.
8. **mojibake (تلف ترميز)** داخل تعليقات الكود في 14 ملفاً رغم توثيق الترميز المكثّف.

## 1.6 فرص التطوير

- تحويل DDL إلى **نظام Migrations منضبط** خارج مسار الطلب.
- **تفكيك الملفات العملاقة** إلى طبقات (Controller / Service / Repository / View).
- **إدارة أسرار** عبر متغيرات بيئة (`.env`) وإزالة الاعتماد من التاريخ.
- **توحيد طبقة الوصول للبيانات** على Prepared Statements.
- إضافة **بحث شامل، إشعارات، لوحات BI، تطبيق/PWA للهاتف**.
- إضافة **API موحّد** لفتح النظام للتكامل ولوحات العملاء/الموردين.

# القسم الثاني: المشاكل البرمجية

> درجات الخطورة: 🔴 حرجة | 🟠 عالية | 🟡 متوسطة | 🟢 منخفضة.

### المشكلة #1 — تنفيذ DDL أثناء معالجة الطلب (Runtime Schema Migration)
- **الخطورة:** 🔴 حرجة
- **الملفات المتأثرة:** `Equipments/equipments.php`، `Clients/clients.php`، `Drivers/drivers.php`، `Contracts/contracts.php`، `Projects/projects.php`، `Suppliers/suppliers.php`، `Oprators/oprators.php`، `main/users.php`، `main/project_users.php`، `movement/movement_operations.php`، `movement/move_oprators.php`، `movement/project_drivers.php`، `Equipments/equipments_drivers.php` (13+ ملفاً).
- **الوصف:** كل صفحة تنفّذ `ALTER TABLE ... ADD COLUMN` و`CREATE INDEX` و`UPDATE` تصحيحية عند كل تحميل (مثل: `ALTER TABLE equipments ADD COLUMN company_id ...`).
- **التأثير:** (1) أداء — فحص/تعديل بنية على كل طلب. (2) **سباق (Race Condition)** عند طلبات متزامنة على نفس الجدول. (3) يفرض منح صلاحية `ALTER` لمستخدم التطبيق في قاعدة البيانات = **سطح خطر أمني كبير**. (4) انجراف المخطط (Schema Drift) بين البيئات. (5) أعطال متقطّعة يصعب تشخيصها.
- **الحل المقترح:** نقل كل تعديلات البنية إلى ملفات Migration تُشغَّل **مرة واحدة عند النشر** عبر CLI، وإزالة جميع استدعاءات DDL من مسار الطلب، وسحب صلاحية ALTER من مستخدم التطبيق في الإنتاج.

### المشكلة #2 — بيانات اعتماد قاعدة بيانات مكشوفة في الكود
- **الخطورة:** 🔴 حرجة
- **الملف المتأثر:** `config.php` (الأسطر 146–154).
- **الوصف:** بيانات اتصال الإنتاج (المضيف، المستخدم، **كلمة المرور كنص صريح**) موجودة كتعليق داخل الملف، وبيانات التطوير (`root` بدون كلمة مرور) مفعّلة. كل ذلك محفوظ في تاريخ Git.
- **التأثير:** أي وصول للمستودع (أو لنسخة قديمة) = تسريب كامل لقاعدة بيانات الإنتاج.
- **الحل المقترح:** نقل الإعدادات إلى `.env` غير متعقّب في Git، استخدام `getenv()`، **تدوير (Rotate) كلمة المرور المسرّبة فوراً**، وتنظيف تاريخ Git (BFG / filter-repo).

### المشكلة #3 — ملفات عملاقة أحادية المسؤولية (God Files)
- **الخطورة:** 🟠 عالية
- **الملفات المتأثرة:** `equipments_fleet.php` (2623)، `Timesheet/timesheet.php` (2612)، `Suppliers/supplierscontracts.php` (2217)، `Drivers/drivercontracts_details.php` (2196)، `equipments_drivers.php` (2050)، `equipments.php` (2004)، وعشرات غيرها فوق 1300 سطر.
- **الوصف:** الملف الواحد يحتوي منطق التفويض + معالجة POST + استعلامات SQL + HTML + CSS مضمّن + JavaScript مضمّن.
- **التأثير:** استحالة المراجعة الآمنة، تعارضات دمج (Merge Conflicts)، صعوبة اختبار، إبطاء التطوير، تكرار المنطق.
- **الحل المقترح:** فصل الطبقات تدريجياً: استخراج SQL إلى Repositories، المنطق إلى Services، العرض إلى قوالب، وتقليل حجم الملف الواحد إلى < 400 سطر.

### المشكلة #4 — بناء SQL بالتسلسل النصّي بدل Prepared Statements
- **الخطورة:** 🟠 عالية
- **الأدلة:** 766 استدعاء `mysqli_query` خام مقابل 139 Prepared Statement فقط؛ 140 موضعاً يحتوي تداخل متغيّرات داخل نص الاستعلام.
- **الوصف:** الاعتماد على `intval()`/`escape()` يدوياً. الأغلب آمن حالياً (أعداد صحيحة)، لكن النمط هشّ: أي إضافة مستقبلية لمدخل نصّي غير مهروب = SQL Injection.
- **التأثير:** خطر حقن SQL كامن + صعوبة تدقيق.
- **الحل المقترح:** توحيد كل الوصول عبر دالة `db_query()` الموجودة (Prepared) أو طبقة Repository، ومنع التسلسل النصّي في مراجعات الكود.

### المشكلة #5 — تلف الترميز (Mojibake) داخل الكود المصدري
- **الخطورة:** 🟡 متوسطة
- **الملفات المتأثرة:** 14 ملف PHP تحتوي تعليقات تالفة مثل `ðŸ" التحقق`.
- **الوصف:** رغم وجود توثيق مكثّف لمنع الترميز التالف، فإن الكود نفسه يحتوي تعليقات تالفة، والحل المتبع هو فلتر إخراج (`ems_fix_mojibake_output`) يفحص **كامل HTML على كل طلب** بتعابير نمطية.
- **التأثير:** فلتر الإخراج عبء أداء على كل صفحة + قد يفسد محتوى صحيحاً (تصحيح زائد False Positive)، والتعليقات التالفة تربك المطوّرين.
- **الحل المقترح:** إصلاح مصدر الترميز (حفظ الملفات UTF-8 بدون BOM)، وإزالة فلتر الإخراج بعد ضمان صحة المصدر.

### المشكلة #6 — أرقام سحرية للأدوار (Magic Numbers)
- **الخطورة:** 🟡 متوسطة
- **الأدلة:** `role == "10"` (24 موضعاً)، `role === '-1'` (54)، `role == "3"/"5"/"6"...` منتشرة.
- **الوصف:** الأدوار مُرمّزة كأرقام نصّية حرفية في كل أنحاء النظام بدل ثوابت/تعداد.
- **التأثير:** تغيير الأدوار خطر، أخطاء تفويض صامتة، صعوبة فهم المنطق.
- **الحل المقترح:** تعريف ثوابت (`ROLE_SUPER_ADMIN = -1`, `ROLE_PROJECT_USER = 10` ...) ومركزة منطق الأدوار.

### المشكلة #7 — تكرار session_start مزدوج
- **الخطورة:** 🟡 متوسطة
- **الوصف:** الوحدات تستدعي `session_start()` مباشرةً ثم `include config.php` الذي يستدعي `secure_session_start()` بإعدادات مشدّدة — فالجلسة قد تبدأ بإعدادات افتراضية قبل المشدّدة.
- **التأثير:** فقدان بعض ضوابط أمان الجلسة (cookie params) في الصفحات التي تبدأ الجلسة مبكراً.
- **الحل المقترح:** إزالة `session_start()` المباشر من الوحدات والاعتماد حصرياً على `secure_session_start()` عبر `config.php`.

### المشكلة #8 — نطاق المستأجر عبر استعلامات فرعية مترابطة هشّة
- **الخطورة:** 🟠 عالية
- **الوصف:** عزل الشركة يتم عبر `EXISTS(SELECT 1 FROM users su WHERE su.id = project.created_by AND su.company_id = $companyId)` بدل عمود `company_id` مباشر على `project`.
- **التأثير:** (1) أداء — استعلام فرعي مترابط متكرر. (2) **هشاشة منطقية**: إذا حُذف/أُعيد تعيين منشئ المشروع تنكسر الرؤية. (3) تكرار نفس المنطق في `dashboard.php` ومعظم الوحدات.
- **الحل المقترح:** إضافة `company_id` مباشر مفهرس على كل جدول رئيسي وملؤه مرة واحدة، ثم تبسيط النطاق إلى `WHERE company_id = ?`.

### المشكلة #9 — غياب التخزين المؤقت لفحص الصلاحيات (N+1)
- **الخطورة:** 🟡 متوسطة
- **الملف:** `includes/permissions_helper.php` — `check_permission()` يُنفّذ Prepared Statement على `role_permissions` في **كل استدعاء**.
- **التأثير:** صفحة تستدعي الفحص عدة مرات = عدة رحلات لقاعدة البيانات لكل تحميل.
- **الحل المقترح:** تحميل كل صلاحيات الدور مرة واحدة في الجلسة/الذاكرة عند الدخول واستعلامها محلياً.

### المشكلة #10 — كود ميت ومكرر (Dead/Duplicate Code)
- **الخطورة:** 🟡 متوسطة
- **الأدلة:** `download_clients_template.php` و`import_clients_excel.php` مكرّران داخل `Projects/`؛ `get_project_mines.php` و`get_mine_contracts.php` مكرّران في 5+ وحدات؛ ملفات `test_modals.html`، `check_rules_table.php`، سكربتات `.tmp_*.ps1` في الجذر.

### المشكلة #11 — ملف بيانات ضخم (12MB) داخل المستودع
- **الخطورة:** 🟢 منخفضة (لكن مزعجة)
- **الملف:** `database/timesheet_30k (2).sql` (~12 ميجابايت).
- **التأثير:** تضخّم المستودع، استنساخ بطيء، احتمال احتواء بيانات حقيقية.
- **الحل المقترح:** إزالته من Git ونقله لتخزين منفصل، وإضافته إلى `.gitignore`.

### المشكلة #12 — CSP يسمح بـ unsafe-inline و unsafe-eval
- **الخطورة:** 🟡 متوسطة
- **الملف:** `includes/security.php` (السطر 446).
- **الوصف:** السياسة تسمح `script-src 'self' 'unsafe-inline' 'unsafe-eval'` لأن السكربتات والأنماط مضمّنة داخل الصفحات.
- **التأثير:** يُضعف الحماية من XSS بشكل جوهري (الفائدة الأساسية من CSP تُلغى).
- **الحل المقترح:** بعد فصل JS/CSS إلى ملفات خارجية، إزالة `unsafe-inline`/`unsafe-eval` واستخدام nonce/hash.

### المشكلة #13 — غياب الفهارس على بعض أنماط النطاق
- **الخطورة:** 🟡 متوسطة
- **الوصف:** بينما يوجد 152 فهرساً، فإن أعمدة الربط في الاستعلامات الفرعية (`created_by`, `company_client_id`) واستعلامات `CAST(o.project_id AS UNSIGNED)` تكسر استخدام الفهارس.
- **التأثير:** فحوص جداول كاملة (Full Scan) عند النمو.
- **الحل المقترح:** توحيد أنواع الأعمدة (إزالة `CAST`)، وإضافة فهارس مركّبة على أعمدة النطاق الفعلية.

### المشكلة #14 — معالجة أخطاء صامتة (`@` Suppression)
- **الخطورة:** 🟡 متوسطة
- **الوصف:** استخدام واسع لعامل كتم الأخطاء `@mysqli_query(...)`، فتُبتلع الأخطاء دون تسجيل سياقي.
- **التأثير:** أعطال صامتة وصعوبة تشخيص، وفقدان سلامة البيانات دون إنذار.
- **الحل المقترح:** إزالة `@`، والتحقق من نتيجة كل استعلام، وتسجيل الأخطاء بسياق.

> مرتّبة حسب: (1) أعلى قيمة تجارية، (2) أعلى أثر على المستخدم، (3) أعلى أثر على الأداء، (4) أعلى أثر على استقرار النظام.

1. تدوير كلمة مرور قاعدة بيانات الإنتاج المسرّبة فوراً.
2. نقل بيانات الاتصال إلى `.env` وإزالتها من الكود.
3. تنظيف تاريخ Git من الأسرار (BFG / filter-repo).
4. سحب صلاحية ALTER من مستخدم قاعدة بيانات الإنتاج.
5. بناء نظام Migrations وتشغيله مرة واحدة عند النشر.
6. إزالة كل استدعاءات `ALTER TABLE`/`CREATE INDEX` من مسار الطلب.
7. فرض فحص صلاحية الوحدة على كل معالجات `*_handler.php`.
8. التحقق من ملكية السجل للشركة في كل معالج AJAX.
9. منع وصول مستخدم «العرض» لإجراءات التعديل/الحذف عبر AJAX.
10. تخزين صلاحيات الدور في الجلسة عند الدخول.
11. تعريف ثوابت الأدوار واستبدال الأرقام السحرية.
12. إزالة `session_start()` المزدوج والاعتماد على الجلسة الآمنة.
13. حذف ملف `timesheet_30k (2).sql` (12MB) من المستودع.
14. حذف الملفات المؤقتة/التجريبية (`.tmp_*.ps1`, `test_modals.html`).
15. إضافة عمود `company_id` مباشر مفهرس للجداول الرئيسية وملؤه.
16. تبسيط منطق النطاق إلى `WHERE company_id = ?`.
17. إزالة `CAST(... AS UNSIGNED)` وتوحيد أنواع الأعمدة.
18. إضافة فهارس على `created_by` و`company_client_id`.
19. توحيد استعلامات COUNT في لوحة التحكم في استعلام مجمّع.
20. إزالة عامل كتم الأخطاء `@` وتسجيل أخطاء الاستعلامات.
21. توحيد الوصول للبيانات عبر `db_query()` المُعدّة (Prepared).
22. بدء طبقة Repository للوحدات الأعلى استخداماً.
23. تفكيك `equipments_fleet.php` (2623 سطراً) كنموذج مرجعي.
24. تفكيك `Timesheet/timesheet.php` (2612 سطراً).
25. استخراج Modals إلى مكوّن موحّد قابل لإعادة الاستخدام.
26. فصل CSS/JS المضمّن إلى ملفات خارجية.
27. تشديد CSP بإزالة `unsafe-inline`/`unsafe-eval`.
28. إصلاح mojibake في المصدر وإزالة فلتر إخراج الترميز.
29. توحيد سياسة الحذف الناعم (Soft Delete) عبر الجداول.
30. إضافة فحص صلاحية على كل تقرير.
31. مركزة منطق نطاق المشروع (role=="10") في دالة واحدة.
32. إضافة بحث شامل (Global Search) أعلى النظام.
33. تحديث الجداول (تجميد رأس + ترقيم + فرز خادمي).
34. إضافة إشعارات أساسية (Toasts/داخلية).
35. إضافة لوحة تحكم تفاعلية برسوم بيانية.
36. إضافة المصادقة الثنائية (2FA) للمستخدمين الإداريين.
37. فرض تغيير كلمة المرور عند أول دخول.
38. توحيد نموذج العقود الثلاثة في نموذج واحد بنوع.
39. بناء محرك Workflow مركزي للموافقات.
40. تعميم سجل التدقيق (Audit Trail) على كل الكيانات.
41. إعداد CI (Lint + اختبارات دخان) في `.github`.
42. كتابة اختبارات لمسارات الدخول/الصلاحيات/العزل.
43. توثيق نموذج البيانات والأدوار رسمياً.
44. إزالة الكود المكرر (`get_project_mines.php` وغيره).
45. تحسين تجاوب الهاتف (بطاقات بدل جداول).
46. إضافة مُنشئ تقارير قابل للتخصيص + تصدير موحّد.
47. تصميم وبناء API عام موثّق (REST).
48. إضافة باركود/QR لجرد المعدات الميداني.
49. إضافة الصيانة الوقائية المجدولة وعدّاد ساعات المعدة.
50. التخطيط لتكامل GPS/Telematics وأولى ميزات التنبؤ بالأعطال.

## خاتمة

النظام يمثّل **منتجاً وظيفياً واسع التغطية** لقطاع متخصّص (إدارة معدات التعدين)، مع **أساس أمني أفضل من المتوسط**. غير أن **الديون المعمارية الحرجة** — وعلى رأسها تنفيذ تعديلات بنية قاعدة البيانات أثناء التشغيل، وتسريب بيانات الاعتماد، والملفات العملاقة، وغياب الاختبارات — تجعل التوسّع لآلاف العملاء **محفوفاً بالمخاطر بشكله الحالي**.

التوصية النهائية: **تجميد إضافة الميزات الكبيرة مؤقتاً**، وتخصيص الشهر الأول بالكامل لإغلاق المهام من 1 إلى 14 أعلاه (الأمان والاستقرار)، ثم الانتقال إلى إعادة الهيكلة التدريجية وفق خارطة الطريق. بهذا التسلسل يمكن رفع جاهزية الإنتاج من **48/100** إلى نطاق **80+/100** خلال 6–9 أشهر.

> *أُعدّ هذا التقرير بمنهجية تدقيق محايدة وصريحة بهدف دعم اتخاذ القرار، لا المجاملة.*
