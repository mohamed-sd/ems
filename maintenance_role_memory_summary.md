# ملخّص النقاش: إضافة دور «مسؤول الصيانة» إلى نظام EMS
> ملف ذاكرة مرجعي — يلخّص كامل الحوار حتى لحظة ما قبل مرحلة الحسم/القرارات.
> التاريخ: 2026-06-20 · النظام: `C:\wamp64\www\ems` (PHP/MySQL, WAMP) · مرفق المواصفات: `08_مواصفات_الصيانة-2.docx`

---

## 1) الهدف
إضافة قسم الصيانة كـ **دور جديد** باسم «مسؤول الصيانة / ادارة الصيانة» على غرار «مسؤول المبيعات»، مع:
- أي معدة تدخل الصيانة → حالتها «تحت الصيانة»، وعند الخروج → «جاهزة/متاحة للعمل».
- ظهور كل تفاصيل الصيانة كتقرير داخل **كرت المعدة** يرافقها أينما عُرض الكرت.

## 2) فهم النظام الحالي (المعمارية)
نظام الأدوار **مبني على بيانات لا كود** عبر ثلاثة جداول:

| الجدول | الدور | أعمدة مفتاحية |
|---|---|---|
| `roles` | تعريف الدور | id, name, parent_role_id, level (1=مدير/2=مشرف), role_scope (gloable/mine), status |
| `modules` | شاشات كل دور | name, code (مسار الصفحة), owner_role_id, is_link, icon, display_order |
| `role_permissions` | صلاحيات الدور على الشاشة | role_id, module_id, can_view/add/edit/delete |
| `users` | ربط المستخدم بالدور | العمود `role` (varchar) يخزّن **رقم الدور** |

- القائمة الجانبية ديناميكية: `includes/dynamic_nav.php` → `getDynamicNavLinks()` تعرض modules حسب `owner_role_id` = دور المستخدم (أو الأب).
- حماية الصفحات: `includes/permissions_helper.php` → `get_page_permissions()` / `enforce_current_page_view_permission()`.
- شاشة المستخدمين `main/users.php` تعبّئ قائمة الأدوار من `roles` تلقائياً ⇒ الدور الجديد يظهر فوراً.
- إدارة الأدوار/الشاشات/الصلاحيات: `admin/permissions/roles.php` · `modules.php` · `role_permissions.php`.
- الدخول `login.php` يوجّه الجميع إلى `main/dashboard.php` (لا تفريع حسب الدور).

### حقائق بيانات مهمة
- الأدوار الحالية (seed): 1 تشغيل، 2 موردين، 3 أسطول، 4 موارد بشرية، 5 مدير موقع (mine)، 6 حركة (mine)، 12 **مبيعات**.
- **النموذج المرجعي (المبيعات):** دور `id=12` + شاشات `modules` 35/36/37 بـ owner_role_id=12 + مستخدم `users.role='12'`. أُضيف ببيانات فقط، بلا كود جديد.
- جداول `roles`/`modules`/`role_permissions` **بلا `company_id`** ⇒ عامة عبر كل الشركات. بينما `users` فيها company_id.
- الترميز: `equipments` = `utf8_general_ci` بينما `modules`/`timesheet_failure_hours` = `utf8mb4_unicode_ci`.

## 3) ملخّص مواصفات الصيانة (المرفق)
منطق Odoo/ERP، إصدار 3.1. ثلاثة خطوط دفاع: عناية يومية ← تفتيش/زيارات/وقائية ← بلاغ/إصلاح. الصيانة **تقرأ** من الإدارات المالكة وتكتب أحداثها فقط.
- الشاشات 8.1–8.9: العناية اليومية، التفتيش، الزيارة الميدانية، البلاغ، **أمر الصيانة** (المحور)، الخطة الوقائية، سجل صيانة المعدة، البيانات المرجعية، مخرجات الإغلاق.
- مؤشرات: MTBF, MTTR, availability_rate, rework_rate, daily_care_compliance...
- المواصفات افترضت وجود: مخزون/شراء، موارد بشرية برواتب، نقل، محرّك اعتماد.

## 4) تطوّر التصميم (القرارات حتى الآن)
1. **الإسقاط على نمط المبيعات**: دور + شاشات + جداول + صلاحيات + مستخدم.
2. **اختصار أول**: 8 شاشات/~16 جدول → 5 شاشات/13 جدول (دمج الزيارة بالتفتيش، سجل المعدة كتقرير، الموافقات بمحرّك قائم، الكتالوجات شاشة واحدة).
3. **اختصار ثانٍ** (مع الحفاظ على الجودة): → **4 شاشات + 8 جداول**.
4. **إزالة الاعتماد نهائياً** (قرار المستخدم): حذف حقول approval، إلغاء مرحلة «بانتظار الاعتماد».
5. **ERD** مرسوم ومحفوظ: `maintenance_erd.mermaid`.

### التصميم القائم حالياً
**الأدوار:** «ادارة الصيانة» (level 1) + «مشرف صيانة» (فرعي اختياري).

**الشاشات (4 + تقرير):**
| الشاشة | code | المحتوى |
|---|---|---|
| أوامر الصيانة | `Maintenance/orders.php` | بلاغ (intake) ← تنفيذ ← فحص ← إغلاق |
| التفتيش الفني | `Maintenance/inspections.php` | معاينة + زيارة ميدانية (نوع) → الجاهزية الفنية |
| الخطة الوقائية | `Maintenance/preventive_plans.php` | جدولة بالساعات/الزمن، تولّد أوامر |
| إعدادات الصيانة | `Maintenance/master_data.php` | كتالوجات + failure_codes القائم |
| تقرير صيانة المعدة + مؤشرات | ضمن `emsreports` | VIEW على mnt_order |

**الجداول الجديدة (8):** `mnt_order`, `mnt_order_labor`, `mnt_order_part`, `mnt_inspection`, `mnt_inspection_line`, `mnt_plan`, `mnt_plan_task`, `mnt_lookup` (+ `mnt_parts` اختياري). كلها بـ id/company_id/code، utf8mb4، وربط بـ `equipments` عبر id رقمي.

**دورة حياة أمر الصيانة (بلا اعتماد):** بلاغ → تنفيذ → فحص → إغلاق (+ ملغى). شرط الإغلاق: actions_taken + root_cause + inspection_result=ناجح ← يعيد المعدة «متاح» ويحدّث آخر صيانة ويعيد جدولة الوقائية ويكتب سجلاً.

## 5) سيناريو حالة المعدة + الكرت (الجاهزية ~85%)
ما هو **جاهز فعلاً** في النظام:
- `equipments.availability_status` فيه أصلاً قيمة **«تحت الصيانة»** و«متاحة للعمل»/«قيد الاستخدام»، + `availability_state` (متوفرة/غير متوفرة) + `last_maintenance_date`.
- الحالة تُقرأ في: `equipments_fleet.php`, `movement/map_page.php`, `movement/move_oprators.php`, `Oprators/oprators.php` ⇒ ضبطها على «تحت الصيانة» ينعكس تلقائياً.
- فلتر اختيار المعدات المتاحة يستبعد ما ليس «متاحة/قيد الاستخدام» ⇒ الاستبعاد من الإسناد تلقائي.
- كرت المعدة = `Equipments/equipment_profile.php` (يُفتح بـ ?id=)، يجمّع التشغيل/السائقين/الساعات ⇒ إضافة تبويب «الصيانة» تتبع نفس النمط.

التطبيق: فتح الأمر → UPDATE availability_status='تحت الصيانة' + state='غير متوفرة'؛ الإغلاق → إعادة للحالة السابقة + تحديث last_maintenance_date.

## 6) المخاطر/الأسئلة المكتشفة (22 بنداً) — تنتظر الحسم

### تعارض المسارات والحالة (الأخطر)
1. مسار «إيقاف للصيانة» **قائم مسبقاً** في `move_oprators.php` (يضع 'موقوفة للصيانة' + operations.status=3 **عبر محرّك الموافقات**) → من يملك دخول الصيانة؟
2. تناقض مع إلغاء الاعتماد: المسار القائم يستخدم approval_workflow → نلغيه أم نبقيه؟
3. تعدّد قيم نفس الحالة: 'موقوفة للصيانة' مقابل 'تحت الصيانة' (يوحّدها `normalize_*` في equipments_fleet.php) → أي كود جديد يمرّ بنفس التطبيع.
4. استعادة الحالة السابقة + من يعيد operations.status المعلّق؟ (نحتاج pre_maintenance_status).

### ازدواج البيانات
5. نظام أعطال قائم وغني: `failure_codes` + `timesheet_failure_hours` + `fleet_failures.php` → نعيد استخدامه أم نكرّره؟
6. مصدر ساعات الوقائية: equipments.operating_hours vs operations.total_equipment_hours vs timesheet.operator_hours → مصدر واحد موثوق.
7. ازدواج الحالة الفنية: المعدة فيها equipment_condition/engine_condition → نتيجة التفتيش تحدّثها لا توازيها.

### تعدد الشركات والوصول
8. الأدوار/الشاشات عامة عبر كل الشركات (بلا company_id) → الدور يُضاف لكل المستأجرين دفعة واحدة.
9. وصول مستخدم الصيانة للكرت: equipment_profile.php متاح للجميع، لكن قائمة الأسطول مملوكة للأسطول → الصيانة لا تستطيع تصفّح المعدات → نحتاج مدخل/قائمة معدات للصيانة.

### تقنية
10. اختلاف الترميز utf8 vs utf8mb4 → جداول الصيانة utf8mb4 + ربط بمفاتيح رقمية.
11. ملكية المعدة (مورد مقابل شركة) `equipments.suppliers` → من يتحمّل التكلفة/يُدخلها الصيانة؟
12. كرت المعدة له دورة اعتماد خاصة (`card_state='active'`, card_approved_by/at في `approve_card.php`) → تعايشها مع availability_status.
13. تصنيف الأعطال القائم يغطّي **3 أنواع معدات فقط** (1=حفار،2=قلاب،3=خرامة) → غير كافٍ لكل الأنواع.
14. لا بنية إشعارات/مهام (فقط `messages`) → كيف يُبلَّغ الفني بالإسناد؟
15. توقيت التوقّف مربوط باعتماد التايم‌شيت (`timesheet_approvals`) → تأخّر احتساب المؤشرات.
16. فهرسة/أداء: mnt_order/mnt_equipment_log تحتاج فهارس (equipment_id, company_id, state, event_date).
17. اتساق الحذف الناعم (is_deleted/deleted_at/deleted_by).
18. نمط الأمان مختلط → التزام prepared statements في شاشات الصيانة.
19. ترحيل بيانات قائمة (معدات حالتها 'موقوفة للصيانة' الآن).
20. تسجيل تقارير الصيانة سهل (report_role_permissions) لكن محتوى التقرير يُبنى يدوياً في emsreports.
21. وراثة الدور الفرعي = **قائمة فقط لا صلاحيات** (dynamic_nav يورّث الروابط، لكن permissions_helper يتحقق بـ role_id المطابق فقط) → «القائمة ظاهرة والوصول مرفوض» ما لم تُنسخ صلاحيات الأب للفرعي.
22. الهبوط بعد الدخول موحّد على dashboard عام → مستخدم الصيانة يرى مؤشرات غير متعلقة ما لم نضف بطاقات صيانة.

**الخلاصة:** سطح المخاطر المعماري **مُستنفَد** عند 22 بنداً؛ ما بعده تفاصيل تنفيذ. أهمّ القرارات: (أ) من يملك تحويل حالة المعدة، (ب) مصير مسار move_oprators واعتماده، (ج) إعادة استخدام failure_codes أم لا.

## 8) القرارات المعتمدة (من نقاش الأسئلة — 16 سؤالاً عبر 4 دفعات)

| # | القرار المعتمد |
|---|---|
| DEC-01 | **عزل لكل شركة (الدور ليس عاماً):** لكل شركة قسم صيانتها الخاص. `company_id` **NOT NULL إجباري** على كل جداول الصيانة وكل استعلام مقيّد به. (صف تعريف الدور يبقى في `roles` كنمط النظام، والعزل الفعلي على مستوى البيانات). *(محدّث بناءً على طلب المستخدم)* |
| DEC-02 | ننشئ **دورين: «مدير الصيانة» + «مشرف صيانة»**. **المدير = كل الصلاحيات**؛ **المشرف = العرض فقط** (`can_view=1` والباقي 0). صفوف `role_permissions` صريحة لكليهما (الوراثة للقائمة فقط لا الصلاحيات). *(محدّث)* |
| DEC-03 | **البلاغ شاشة موحّدة لكل الإدارات** على نمط شاشة الرسائل/المحادثات (تتجاوز فحص الصلاحية)، **أيقونة في التوبار**. البلاغ = تذكرة/فورم شامل فقط، **لا يغيّر حالة المعدة**. |
| DEC-04 | سير العمل: مسؤول الصيانة يراجع البلاغ → يُصدر أمر صيانة. **إشعار البلاغات = قائمة «جديدة» + عدّاد** (بلا بنية إشعارات). |
| DEC-05 | **دخول المعدة «تحت الصيانة» حصري لمدير الحركة والتشغيل** كما في النظام الحالي (`move_oprators` يبقى). الصيانة لا تُدخل المعدة الصيانة. |
| DEC-06 | **خروج الصيانة: إغلاق أمر الصيانة يعيد المعدة «متاحة للعمل» تلقائياً** (الموضع الوحيد الذي تكتب فيه الصيانة الحالة). |
| DEC-07 | **إعادة استخدام `failure_codes` كما هو** (FK = id فقط، بلا تعديل بياناته)، و**نقل شاشة إدارته `manage_failure_codes.php` من الأسطول إلى الصيانة** (حصراً لمدير/موظفي الصيانة). |
| DEC-08 | **ساعات الوقائية من التايم‌شيت الفعلية** (`timesheet.operator_hours`). |
| DEC-09 | **توليد الوقائية: قائمة «مستحقة» + زر توليد يدوي** (بلا cron). |
| DEC-10 | **قطع الغيار: إدخال يدوي** (اسم + تكلفة) في سطر الأمر، بلا مخزون ← **حذف جدول `mnt_parts`**. |
| DEC-11 | **الصيانة تغطّي كل المعدات** (شركة ومورد)، مع حقل **«جهة التكلفة»** (داخلي/خارجي، شركة/مورد). |
| DEC-12 | **نتيجة التفتيش تُحدّث حقول المعدة** (`equipment_condition`/`engine_condition`) على الكرت + تُخزَّن في الصيانة. الزيارة الميدانية = نوع تفتيش مدموج. |
| DEC-13 | **تبويب «الصيانة» في الكرت مرئي لكل من يصل للكرت** (قراءة). |
| DEC-14 | **لا اعتماد في الصيانة نهائياً** (دورة الأمر: بلاغ→تنفيذ→فحص→إغلاق). محرّك الموافقات يبقى لمسار التشغيل `move_oprators` فقط، خارج نطاق الصيانة. |
| DEC-15 | **معايير تقنية:** جداول الصيانة `utf8mb4_unicode_ci`، ربط بـ `equipments` عبر **id رقمي**، نمط حذف ناعم (is_deleted)، prepared statements، فهارس (equipment_id/company_id/state). |
| DEC-16 | **(افتراض للمراجعة)** تكلفة العمالة **إدخال يدوي** (ساعات + تكلفة) لعدم وجود رواتب في HR — مثل قطع الغيار. مؤشرات النسخة الأولى: التوقّف، MTBF، MTTR، التكلفة، نسبة الجاهزية. |
