## الهدف العام

ابنِ نظاماً متكاملاً من جزأين على نظام EMS الحالي (PHP + MySQL، عربي RTL، متعدّد الشركات):

1. **طبقة REST API** (PHP) تعمل على نفس قاعدة بيانات النظام وتغطّي عمليات **مدير الموقع (التايم شيت)** + **المزامنة (Sync)**.
2. **تطبيق Flutter كامل وجاهز يعمل دون اتصال (Offline-First)** يستهلك هذه الـ API، مطابق لملف التصميم المرفق.

التطبيق مخصّص لدور **مدير الموقع** (الدور 5، نطاقه «mine» = مقيّد بمشروع واحد). المشروع **يُعرف تلقائياً من حسابه عند الدخول** (`project_id` في جدول `users`). لا توجد شاشة اختيار مشروع. مهمته: **توثيق ومتابعة ساعات عمل المعدات يومياً (التايم شيت)**.

> ملاحظة: قد يوجد في المشروع مسبقاً مجلد `api/` لتطبيق مدير الحركة والتشغيل. أعِد استخدام بنيته (المصادقة بالتوكن، تنسيق الردود، معالجة الأخطاء) وأضِف نقاط مدير الموقع دون كسر ما هو قائم.

## المصادر التي يجب قراءتها أولاً (مصدر الحقيقة لمنطق العمل)

اقرأ هذه الملفات بالكامل قبل أي كتابة، فالـ API يجب أن **تطابق منطقها وقواعدها واستعلاماتها** بدقّة:

- `Timesheet/timesheet.php` — منطق إدخال/تعديل/حذف التايم شيت، الحقول حسب النوع، الحسابات التلقائية، تحقّق الأعطال.
- `Timesheet/timesheet_type.php` — اختيار نوع الكشف (1/2/3).
- `Timesheet/view_timesheet.php` — قائمة السجلات والفلاتر والإحصاءات ومنطق مدير الموقع (`role===5`).
- `Timesheet/timesheet_details.php` — تفاصيل سجل مفرد.
- نقاط AJAX: `Timesheet/get_operations.php`, `get_drivers.php`, `get_contract_hours.php`, `get_timesheet_data.php`, `get_timesheet.php`, `get_timesheet_failures.php`, `get_failure_codes.php`.
- `config.php`, `includes/permissions_helper.php`, `login.php` — الاتصال والصلاحيات وبناء الجلسة.
- `BRIEF_flutter_site_manager.md` — الوصف الوظيفي الكامل (المهام + قواعد العمل + ميزة الأوفلاين والمزامنة).
- ملف التصميم المرفق (المسار أعلاه) — المرجع البصري الملزِم للتطبيق.

# الجزء الأول: طبقة الـ API (PHP)

## المبادئ

- ضعها في مجلد **`api/`** (أعِد استخدام الموجود إن وُجد)، **دون كسر** النظام الحالي.
- أعد استخدام اتصال قاعدة البيانات من `config.php` ودوال `db_table_has_column` والصلاحيات وعزل الشركة كما هي.
- **مصادقة Token (Bearer):** عند الدخول يُصدَر توكن في جدول `api_tokens`، ويُرسل عبر `Authorization: Bearer <token>`. لا CSRF في الـ API.
- **كل الاستعلامات Prepared Statements**، وردود JSON موحّدة: `{ "success": bool, "message": string, "data": mixed }` مع `JSON_UNESCAPED_UNICODE` وترويسة `application/json; charset=utf-8`.
- **عزل صارم:** كل endpoint يتحقق من التوكن ويستخرج `user_id, role, company_id, project_id`، ويقصر كل البيانات على **شركة ومشروع** المستخدم. **مهم:** افرض تقييد المشروع حتى على endpoint تفاصيل السجل المفرد (الشاشة الويب القديمة `timesheet_details.php` لا تفرضه — صحّح ذلك في الـ API).
- أكواد HTTP صحيحة ورسائل عربية بنفس أسلوب النظام، ونقطة معالجة أخطاء مركزية تحوّل أي استثناء/خطأ فادح إلى JSON.

## نقاط النهاية المطلوبة (Endpoints)

**المصادقة والسياق:**
- `POST /api/login` — username, password → توكن + بيانات المستخدم (id, name, role, company_id, project_id) + بيانات المشروع.
- `POST /api/logout` · `GET /api/me`.

**بيانات مرجعية للتايم شيت (تُستهلك أونلاين وتُخزَّن محلياً للأوفلاين):**
- `GET /api/timesheet/refdata?updated_since=` — حزمة مرجعية كاملة لمشروع المستخدم تكفي للعمل offline: العمليات (operations) مع معداتها، المعدات، السائقون، ربط المعدات بالسائقين (equipment_drivers)، العقود، أنواع المعدات، و**شجرة أكواد الأعطال كاملة (`failure_codes`)**. ادعم السحب التزايدي عبر `updated_since` (إرجاع المتغيّر فقط) مع طابع زمني للخادم.
- `GET /api/operations/by-type?type=&shift=` — الآليات (operations) حسب نوع الكشف والوردية (مكافئ `get_operations.php`، لكن JSON خام: `[{operation_id, code, name}]`).
- `GET /api/operations/{operation_id}/drivers?shift=` — السائقون المرتبطون بمعدة التشغيل (مكافئ `get_drivers.php`، JSON خام).
- `GET /api/operations/{operation_id}/contract-hours` — `{ shift_hours, shift_type, allowed_shifts }` (مكافئ `get_contract_hours.php`).
- `GET /api/failure-codes?...` — الشجرة الرباعية (نوع الحدث ← الفئة ← الجزء ← التفصيل) JSON خام (مكافئ `get_failure_codes.php`).

**سجلات التايم شيت:**
- `GET /api/timesheets?type=&operation_id=&driver_id=&shift=&date=&start_date=&end_date=&month=&status=` — قائمة سجلات مشروع المستخدم مفلترة، **JSON خام** (لا HTML)، مع الحقول كقيم لبناء النماذج والشارات في التطبيق. أرفِق إجماليات للبطاقات الإحصائية (منفّذة/استعداد/أعطال/إجمالي العمل).
- `GET /api/timesheets/{id}` — تفاصيل سجل واحد (كل الحقول + الأعطال المصنّفة) **ضمن نطاق المشروع**.
- `POST /api/timesheets` — إضافة سجل: الحقول حسب النوع (انظر القواعد). يطبّق **كل التحقّقات والحسابات** كما في `timesheet.php`. يقبل **قائمة أعطال متعددة** ويحفظها في `timesheet_failure_hours` (داخل Transaction).
- `PUT /api/timesheets/{id}` — تعديل سجل (نفس القواعد).
- `DELETE /api/timesheets/{id}` — حذف سجل.

**المزامنة (Sync) — محورية للأوفلاين:**
- `POST /api/sync/timesheets` — **رفع دفعي (batch)** لمجموعة تغييرات معلّقة من الجهاز. كل عنصر يحمل: `op` (create/update/delete)، `client_uuid` (معرّف محلي فريد)، `payload`، و`client_updated_at`. الردّ يعيد لكل عنصر: `{ client_uuid, status (applied/conflict/error), server_id, message }`. **idempotent** عبر `client_uuid` (لا تُكرّر الإدراج إن وصل مرتين).
- `GET /api/sync/pull?updated_since=` — سحب تزايدي لتغييرات السجلات والبيانات المرجعية منذ آخر مزامنة (يعيد طابع زمن الخادم لاستخدامه في الطلب التالي).
- سياسة التعارض: حدّدها بوضوح ووثّقها (مثل: الأحدث `updated_at` يفوز، أو وضع علامة conflict للمراجعة) وأعِدها في ردّ الرفع.

> طابِق **القواعد الصارمة**:
> - **نوع الكشف (type)** يحدّد الحقول: حفّار(1) ساعات جردل/جاك همر و«المنفّذة محسوبة تلقائياً»؛ قلّاب(2) أوزان/نقلات؛ خرّامة(3) أمتار/حفر/عمق و«الأمتار محسوبة تلقائياً».
> - **تحقّق الأعطال الإلزامي:** مجموع جهات الأعطال = إجمالي ساعات التعطّل بالضبط، وإلا 422.
> - **الحقول المحسوبة تلقائياً** تُحسب في الخادم أيضاً (لا تثق بالعميل وحده): المنفّذة للحفّار، إجمالي العمل، إجمالي الأعطال، الأمتار، فرق العدّاد.
> - الوردية D/N (وبعض العمليات B). حالة السجل تُنشأ «تحت المراجعة» (1).

# الجزء الثاني: تطبيق Flutter (Offline-First)

## المتطلبات

- **نفّذ ملف التصميم المرفق بدقّة**. إن نقص تفصيل وظيفي، عُد إلى `BRIEF_flutter_site_manager.md`.
- مشروع Flutter كامل وجاهز للتشغيل في مجلد جديد (مثل `flutter_site_manager/` — وضّحه)، منفصل عن أي تطبيق Flutter آخر في المستودع.
- **عربي بالكامل، RTL**.

### الشاشات
1. **الدخول** — `/api/login` → حفظ التوكن بأمان (`flutter_secure_storage`) → الدخول مباشرة على مشروع المستخدم. يجب أن يعمل الاستئناف **حتى دون إنترنت** (توكن + بيانات محلية).
2. **اختيار نوع الكشف** (حفّار/قلّاب/خرّامة).
3. **إدخال/تعديل التايم شيت** — نموذج ديناميكي حسب النوع: اختيار متسلسل (الوردية ← الآلية ← السائق ← التاريخ، وتعبئة ساعات الوردية تلقائياً)، الحقول النوعية، قسم الأعطال مع **الشجرة الرباعية** و**قائمة أعطال متعددة**، الحقول المحسوبة كقراءة فقط، وتطبيق **قاعدة تحقّق الأعطال** قبل الحفظ. **الحفظ يعمل offline فوراً.**
4. **قائمة السجلات** — فلاتر (نوع/آلية/سائق/وردية/تاريخ/فترة/شهر/حالة)، بطاقات إحصائية، شارة حالة السجل وشارة **حالة المزامنة** لكل صف، وتصدير عند الاتصال.
5. **تفاصيل سجل**.
6. **لوحة/شاشة المزامنة** + **مؤشر الاتصال** الدائم في الرأس + عدّاد غير المزامَن.

## العمل دون اتصال والمزامنة (Offline-First) — جوهر التطبيق

- **قاعدة بيانات محلية (SQLite — Drift أو sqflite)** تخزّن: البيانات المرجعية (من `/api/timesheet/refdata`) + سجلات التايم شيت + **طابور تغييرات معلّقة**.
- **كل العمليات تعمل محلياً فوراً** (إضافة/تعديل/حذف) بنفس قواعد التحقّق والحسابات، دون انتظار الشبكة.
- كل سجل: **`client_uuid`** محلي + **`server_id`** (بعد المزامنة) + **حالة مزامنة** (`pending`/`synced`/`error`) + `updated_at`.
- **كشف الاتصال تلقائياً** (`connectivity_plus`) ثم مزامنة (تلقائية + زر يدوي): **رفع** الطابور عبر `/api/sync/timesheets` (مع `client_uuid` لمنع التكرار) ثم **سحب** عبر `/api/sync/pull`، وربط `server_id`، ومعالجة التعارضات/الأخطاء.
- **الواجهة:** مؤشر اتصال (متصل/غير متصل)، عدّاد «X بانتظار الرفع»، لوحة مزامنة (آخر مزامنة، تقدّم، قائمة المعلّق، الأخطاء + إعادة محاولة)، وشارة مزامنة لكل سجل، ورسائل مثل «حُفظ محلياً، سيُرفع عند توفّر الإنترنت».

## البنية والجودة

- معمارية نظيفة: API client (Dio) + Local DB (Drift/sqflite) + Repositories تُوحّد المصدرين (محلي/شبكة) + إدارة حالة (Riverpod/Bloc — اختر وبرّر) + شاشات/widgets.
- **عميل API مركزي:** يضيف Bearer تلقائياً، يعالج 401، ويوحّد `{success, message, data}` والأخطاء العربية.
- **مكوّنات قابلة لإعادة الاستخدام:** بطاقة نوع الكشف، حقول الساعات/العدّاد، قائمة الأعطال + الشجرة المتسلسلة، القوائم المتسلسلة، شارات الحالة/المزامنة، مؤشر الاتصال.
- **معالجة الاصطلاحات:** التواريخ `YYYY-MM-DD`؛ ألوان دلالية متّسقة؛ الحقول المحسوبة للقراءة فقط.
- `flutter_site_manager/README.md`: التشغيل، ضبط `base_url`، التبعيات، وشرح آلية المزامنة.

# التحقق (إلزامي قبل الإنهاء)

**للـ API:**
1. `php -l` على كل ملفات `api/` — بلا أخطاء.
2. اختبر (curl/Postman): login → refdata → timesheets (GET/POST/PUT/DELETE) → sync (رفع دفعة بها create/update + إعادة إرسال نفس `client_uuid` للتأكد من idempotency) → pull. تأكّد من: العزل بالمشروع (حتى على تفاصيل السجل)، تحقّق الأعطال (422 عند عدم التطابق)، والحسابات التلقائية.
3. تأكّد أن النظام الويب الحالي وأي API قائمة لم تتأثّر.

**للتطبيق:**
4. `flutter analyze` بلا أخطاء و`flutter build` ينجح.
5. شغّل على محاكي وتتبّع: دخول → تنزيل البيانات المرجعية → **تفعيل وضع الطيران** → إضافة/تعديل تايم شيت بالأنواع الثلاثة offline (مع تحقّق الأعطال والحسابات) → التأكد من حفظها محلياً وظهور عدّاد المعلّق → **إعادة الاتصال** → التحقق من المزامنة التلقائية ورفع السجلات وتحوّل حالتها إلى «مزامَن». تأكّد من RTL والعربية.
