# خارطة طريق: بناء «الأسطول والجاهزية» (EQUIP-OPE-S03) على نظام EMS الحالي (PHP/MySQL)

> تكييف مواصفة Odoo المؤسسية إلى نظام EMS الحالي (PHP خام + MySQLi، عربي RTL، متعدّد الشركات)، على مراحل واقعية بأقل مخاطرة ودون كسر النظام.

---

## 1. مبادئ التكييف (Odoo → PHP EMS)

المواصفة مكتوبة بمفاهيم Odoo؛ نترجمها كالآتي:

| مفهوم Odoo | المقابل في PHP/MySQL |
|---|---|
| نموذج (`equipation.equipment`) | جدول MySQL بادئة `fleet_` |
| Many2one | عمود مفتاح أجنبي (FK) |
| One2many (سطور) | جدول أبناء منفصل |
| Many2many / مرفقات | جدول وصل + مسارات ملفات على القرص |
| Selection | ENUM أو جدول مرجعي |
| (محسوب / computed) | يُحسب في PHP عند القراءة، أو يُجسَّد عبر **cron/مهمة دورية** (الإهلاك الشهري) |
| محرّك سير عمل Odoo | **آلة حالات صريحة في PHP**: ١٥ حالة + حُرّاس انتقال + مخوّل لكل انتقال |
| `ir.sequence` (التكويد S11) | جدول عدّاد للكود الدائم |
| سجل بالإضافة فقط (history) | جدول `fleet_equipment_history` إدراج فقط (لا حذف/تعديل) |

**حوكمة صارمة محمولة كما هي:** كود دائم واحد للمعدة، لا كرت إلا من استمارة معتمدة، فصل الملكية (الأسطول) عن الحالة الفنية (الصيانة) عن المالية، ومصدر حقيقة واحد لكل بيان.

---

## 2. ما يُعاد استخدامه من EMS الحالي (لا يُبنى من الصفر)

- **`equipments`** الحالي → يُرحَّل/يُوسَّع ليصبح **كرت المعدة `fleet_equipment`** (مع الحفاظ على الكود مستقراً). أغلب حقول الكرت موجودة جزئياً.
- **`operations` + `timesheet`** → مصدر **`engine_hours`** (ساعات التشغيل) و**ساعات التوقف** — تُغذّي الكرت (مكافئ L5/L3).
- **`failure_codes` + `timesheet_failure_hours`** → بيانات الأعطال (شريحة من S08).
- **محرّك الحصص** (الذي خططنا له) → الإتاحة المحسوبة + قاعدة «لا تخصيص إلا لمعدة متاحة» + حظر التخصيص عند الصيانة الحرجة.
- **الموردون، المشاريع، المستخدمون، الصلاحيات، عزل الشركة، إطار Excel** → كما هي.

---

## 3. التبعية الحرجة: إدارة الصيانة (S08) غير موجودة

المواصفة تعتمد على S08 في كل مكان (تفتيش الاستلام يبوّب الجاهزية، الخطة الوقائية تعطي الاستحقاق، الأعطال/الأوامر تعطي التكلفة وساعات التوقف). **هذه الوحدة غير موجودة في EMS** (الموجود شريحة أعطال عبر التايم شيت فقط). الخيارات:

- **(أ) بناء وحدة صيانة كاملة أولاً** — يضاعف حجم المشروع.
- **(ب) المُوصى به — جسر مبسّط:** نبني داخل الأسطول **خطوة اعتماد جاهزية فنية مبسّطة** + **سجل عطل/توقف مبسّط** كبديل مؤقت لـ S08، مع **منفذ تكامل واضح (interface seam)** نستبدله بـ S08 حقيقي لاحقاً دون إعادة بناء.

> القرارات المالية أسهل: الأسطول **يحسب الإهلاك بنفسه** (UOP) ويُنتج تقريراً شهرياً (المواصفة: الأسطول يحسب والمالية تُرحّل) — لا نحتاج وحدة مالية كاملة، فقط جدول ملف الافتراضات + cron شهري. والتكويد S11 = عدّاد بسيط.

## 4. مخطّط الجداول الجديد (مبدئي)

- `fleet_model` (سجل النوع والموديل §9.1) + `fleet_model_service_spec` (سطور §9.1-أ).
- `fleet_depreciation_profile` (§9.2، تملكه المالية).
- `fleet_equipment_intake` (استمارة إدخال الأصل §9.3 — بوابة الإنشاء).
- `fleet_equipment` (كرت المعدة §9.4 — توسعة `equipments`).
  - أبناء: `fleet_equipment_compliance` (وثائق)، `fleet_equipment_protection` (حماية)، `fleet_equipment_component` (مكوّنات كبرى).
- `fleet_equipment_history` (تاريخ المعدة §9.6 — إدراج فقط).
- `fleet_equipment_transfer` (الترحيل) · `fleet_equipment_disposal` (الخروج).
- `fleet_code_sequence` (الكود الدائم — S11).
- `fleet_depreciation_line` (سطر الإهلاك الشهري).
- جسر S08 المبسّط: `fleet_tech_inspection` + `fleet_breakdown` (تُستبدل لاحقاً).

كلها بـ `company_id` للعزل، Prepared Statements، و`db_table_has_column` للتوافق.

## 5. المراحل (مرتّبة بالاعتماديات)

**المرحلة 0 — الأسس والمراجع** (تمكّن كل ما بعدها)
- `fleet_model` + service specs + `fleet_depreciation_profile` + عدّاد الكود الدائم.
- توسعة `equipments` → كرت `fleet_equipment` بالحقول الجديدة (المصدر، الأرقام، العداد الافتتاحي، ربط الموديل/الملف المالي). بلا دورة حياة بعد.

**المرحلة 1 — بوابة الدخول وإنشاء الكرت**
- استمارة إدخال الأصل + اعتماد إداري → إنشاء الكرت آلياً (كود دائم + وراثة الموديل والافتراضات + تثبيت العداد الافتتاحي)، الحالة = «قيد الفحص الفني».

**المرحلة 2 — آلة الحالات ودورة الحياة**
- ١٥ حالة + حُرّاس الانتقال + المخوّل لكل انتقال + `fleet_equipment_history` (إدراج فقط).
- جسر الجاهزية الفنية المبسّط (بديل S08 المؤقت) + مجمّع المتاح (إتاحة محسوبة).
- ربط بمحرّك الحصص: لا تخصيص إلا لمتاحة، وتخرج من المتاح تلقائياً عند الصيانة.

**المرحلة 3 — محرّك العدّاد والاستحقاق**
- `engine_hours` تراكمي من التايم شيت/L5 + عرض الاستحقاق + حظر التخصيص عند الصيانة الحرجة (عبر محرّك الحصص).

**المرحلة 4 — الإهلاك (UOP)**
- حساب معدل الساعة والمتراكم والقيمة الدفترية + **cron شهري** يُنتج تقرير الإهلاك + تصدير قيد للحسابات.

**المرحلة 5 — الترحيل والخروج والمؤشرات**
- الترحيل بين المواقع + الخروج/الاستبعاد + تحليلات الأسطول (MTBF/MTTR تُقرأ من S08 عند توفّره).

**موازٍ اختياري — وحدة صيانة S08 حقيقية** تستبدل الجسر المبسّط (تفتيش، خطة وقائية، أوامر، أعطال).

## 6. مخاطر وقرارات يجب الانتباه لها

1. **حجم المشروع:** ٥ مراحل + احتمال وحدة صيانة. ليس «شاشة». خطّط بالمراحل لا دفعة واحدة.
2. **ترحيل بيانات `equipments` الحالية** إلى نموذج الكرت دون فقدان أو كسر الشاشات المرتبطة (التشغيل/التايم شيت تقرأ من `equipments`). نهج آمن: توسعة الجدول الحالي تدريجياً لا استبداله فجأة.
3. **توحيد «نوع المعدة»:** المواصفة توسّع الأنواع (حفّار·قلّاب·لودر·قريدر·دوزر·مولّد·خرّامة·شاحنة·رافعة...) — يجب توحيدها مع `equipments_types` بدل القيم الثابتة 1/2/3 المبعثرة في الكود.
4. **الحقول المحسوبة:** قرار لكل حقل — يُحسب عند القراءة (أرخص) أم يُجسَّد بـ cron (الإهلاك الشهري والمؤشرات).
5. **جسر S08:** صمّم الواجهة (interface) من البداية حتى يكون استبدال الجسر بوحدة صيانة حقيقية لاحقاً غير كاسر.

---

## 7. الخطوة التالية المقترحة

نبدأ بـ **المرحلة 0 (الأسس + مخطّط البيانات + ترحيل آمن لـ`equipments`)** لأنها تفتح كل ما بعدها وأقلّها مخاطرة. لكل مرحلة سننتج: بريف Claude Design للشاشات، ثم برومبت Claude Code للبناء والتحقق.

**القرارات المطلوبة قبل المرحلة 0:**
- جسر S08 المبسّط الآن أم بناء وحدة صيانة كاملة من البداية؟ (المُوصى به: الجسر.)
- الإهلاك ضمن النطاق الآن أم مرحلة لاحقة؟
- هل نوسّع `equipments` الحالي (موصى به) أم ننشئ `fleet_equipment` جديداً ونرحّل؟
