# نظام الاعتماد - دليل الاستخدام والاختبار

## نظرة عامة

تم تنفيذ نظام الاعتماد في مكانين فقط حسب المطلوب:

### 1. **تشغيل وإيقاف الآلية** (Oprators/oprators.php)
- **المُقدّم**: مدير الحركة والتشغيل (role 10)
- **المُعتمد**: مدير الأسطول (role 4) أو الإدارة العليا (role -1)
- **الإجراءات**:
  - إيقاف آلية (deactivate_equipment)
  - إعادة تشغيل آلية (reactivate_equipment)

### 2. **تشغيل وإيقاف المشغل/السائق** (Equipments/equipments.php)
- **المُقدّم**: مدير الحركة والتشغيل (role 10)
- **المُعتمد**: مدير المشغلين (role 3) أو الإدارة العليا (role -1)
- **الإجراءات**:
  - تشغيل مشغل جديد (activate_driver)
  - إيقاف مشغل (deactivate_driver)
  - إعادة تشغيل مشغل (reactivate_driver)

## سير العمل

### مدير الحركة والتشغيل (Role 10)

1. **إيقاف آلية**:
   - الدخول إلى `Oprators/oprators.php`
   - اختيار المشروع
   - الضغط على زر "طلب إيقاف" بجانب الآلية
   - كتابة سبب الإيقاف
   - إرسال الطلب → يتم إنشاء طلب موافقة

2. **تشغيل/إيقاف مشغل**:
   - الدخول إلى `Equipments/equipments.php`
   - الضغط على "إدارة المشغلين" بجانب الآلية
   - **إضافة مشغل جديد**: اختيار المشغل وتواريخ العمل → إرسال طلب موافقة
   - **إيقاف مشغل**: الضغط على زر التبديل بجانب المشغل → إرسال طلب موافقة

### مدير الأسطول (Role 4)

1. الدخول إلى `Approvals/requests.php` من القائمة الجانبية
2. عرض طلبات إيقاف/تشغيل الآليات فقط
3. لكل طلب:
   - **عرض**: عرض تفاصيل الطلب (معلومات الآلية، السبب، إلخ)
   - **اعتماد**: الموافقة على الطلب → تنفيذ التغيير تلقائيًا
   - **رفض**: رفض الطلب → لا يتم تنفيذ التغيير

### مدير المشغلين (Role 3)

1. الدخول إلى `Approvals/requests.php` من القائمة الجانبية
2. عرض طلبات تشغيل/إيقاف المشغلين فقط
3. لكل طلب:
   - **عرض**: عرض تفاصيل الطلب (معلومات المشغل، الآلية، التواريخ، إلخ)
   - **اعتماد**: الموافقة على الطلب → تنفيذ التغيير تلقائيًا
   - **رفض**: رفض الطلب → لا يتم تنفيذ التغيير

### الإدارة العليا (Role -1)

- يمكنها رؤية واعتماد/رفض **جميع الطلبات** (آليات ومشغلين)
- تظهر جميع الطلبات في صفحة `Approvals/requests.php`

## جداول قاعدة البيانات

### approval_workflow_rules
قواعد الموافقة المضافة:

| Entity Type | Action                  | Role Required | Step Order | Active |
|-------------|-------------------------|---------------|------------|--------|
| driver      | activate_driver         | 3,-1          | 1          | 1      |
| driver      | deactivate_driver       | 3,-1          | 1          | 1      |
| driver      | reactivate_driver       | 3,-1          | 1          | 1      |
| equipment   | deactivate_equipment    | 4,-1          | 1          | 1      |
| equipment   | reactivate_equipment    | 4,-1          | 1          | 1      |

### approval_requests
يحتوي على جميع الطلبات المُرسلة (pending/approved/rejected)

### approval_steps
يحتوي على مراحل الموافقة لكل طلب

## آلية العمل التقنية

### 1. إنشاء طلب موافقة

```php
// في delete_equipment_driver.php أو save_equipment_drivers.php
$payload = [
    'summary' => [
        'driver_id' => $driver_id,
        'driver_name' => $driver_name,
        'equipment_id' => $equipment_id,
        'action' => 'إيقاف مشغل',
        // ...
    ],
    'operations' => [
        [
            'db_action' => 'update',
            'table' => 'equipment_drivers',
            'where' => ['id' => $id],
            'data' => ['status' => 0]
        ]
    ]
];

$result = approval_create_request(
    'driver',
    $driver_id,
    'deactivate_driver',
    $payload,
    approval_get_user_id(),
    $conn
);
```

### 2. الاعتماد/الرفض

عند الضغط على "اعتماد" في `Approvals/requests.php`:
- يتم إرسال AJAX request إلى `Approvals/approval_api.php`
- يتم استدعاء `approval_approve_request()`
- يتم تنفيذ الـ operations في الـ payload (UPDATE/INSERT/DELETE)
- يتم تحديث حالة الطلب إلى "approved"

عند الضغط على "رفض":
- يتم استدعاء `approval_reject_request()`
- يتم تحديث حالة الطلب إلى "rejected"
- **لا يتم تنفيذ أي تغييرات في البيانات**

## الاختبار

### سيناريو 1: إيقاف آلية
1. تسجيل دخول بـ role 10 (مدير الحركة والتشغيل)
2. الدخول إلى `Oprators/oprators.php`
3. اختيار مشروع
4. الضغط على "طلب إيقاف" بجانب آلية نشطة
5. كتابة السبب والإرسال
6. **النتيجة المتوقعة**: رسالة نجاح "تم إرسال الطلب للموافقة"

7. تسجيل خروج وتسجيل دخول بـ role 4 (مدير الأسطول)
8. الدخول إلى "طلبات الموافقات"
9. يجب ظهور الطلب مع أزرار "عرض" و "اعتماد" و "رفض"
10. الضغط على "اعتماد"
11. **النتيجة المتوقعة**: رسالة نجاح "تم الاعتماد النهائي وتنفيذ العملية"
12. العودة إلى `Oprators/oprators.php` والتحقق من تغيير حالة الآلية

### سيناريو 2: إضافة مشغل جديد
1. تسجيل دخول بـ role 10
2. الدخول إلى `Equipments/equipments.php`
3. الضغط على "إدارة المشغلين" بجانب آلية
4. اختيار مشغل من القائمة وتحديد تواريخ العمل
5. الضغط على "حفظ"
6. **النتيجة المتوقعة**: رسالة "تم إرسال طلب موافقة لتشغيل المشغلين"

7. تسجيل خروج وتسجيل دخول بـ role 3 (مدير المشغلين)
8. الدخول إلى "طلبات الموافقات"
9. يجب ظهور الطلب
10. الضغط على "عرض" لمراجعة التفاصيل
11. الضغط على "اعتماد"
12. **النتيجة المتوقعة**: إضافة المشغل إلى الآلية

### سيناريو 3: رفض طلب
1. تكرار أحد السيناريوهات السابقة
2. عند الوصول إلى شاشة الاعتماد (role 3 أو 4)
3. الضغط على "رفض" بدلاً من "اعتماد"
4. كتابة سبب الرفض
5. **النتيجة المتوقعة**: رسالة "تم رفض الطلب بنجاح"
6. **التحقق**: لا يتم تنفيذ التغيير على البيانات

## نقاط هامة

### 1. الصلاحيات
- **Role 10**: يُقدّم طلبات فقط، لا يملك صلاحية التنفيذ المباشر
- **Role 3**: يعتمد طلبات المشغلين فقط
- **Role 4**: يعتمد طلبات الآليات فقط
- **Role -1**: يعتمد جميع الطلبات

### 2. الفلترة التلقائية
- صفحة `Approvals/requests.php` تعرض فقط الطلبات ذات الصلة بصلاحية المستخدم
- user role 3 يرى طلبات driver فقط
- user role 4 يرى طلبات equipment فقط
- user role -1 يرى جميع الطلبات

### 3. التنفيذ التلقائي
- عند الاعتماد: يتم تنفيذ الـ operations المحددة في payload تلقائيًا
- عند الرفض: لا يتم تنفيذ أي شيء، فقط تسجيل سبب الرفض

### 4. الأمان
- جميع الطلبات محمية بـ CSRF tokens
- التحقق من الصلاحيات في كل خطوة
- prepared statements لمنع SQL injection

## استكشاف الأخطاء

### المشكلة: لا تظهر الطلبات في صفحة الموافقات
**الحلول**:
1. التحقق من أن الطلب تم إنشاؤه بنجاح (status = pending)
2. التحقق من role_required في approval_steps يتطابق مع role المستخدم
3. تنفيذ `database/update_approval_rules.php` لضمان وجود القواعد

### المشكلة: أزرار الاعتماد/الرفض لا تعمل
**الحلول**:
1. فتح console المتصفح للتحقق من أخطاء JavaScript
2. التحقق من تحميل Bootstrap 5 بشكل صحيح
3. التحقق من CSRF token صالح

### المشكلة: فشل التنفيذ بعد الاعتماد
**الحلول**:
1. التحقق من صحة الـ payload (operations)
2. فحص أخطاء SQL في لوج PHP
3. التأكد من أن اسم الجدول والأعمدة صحيحة

## الملاحظات الفنية

### 1. Auto-seeding للقواعد
الكود يحتوي على auto-seeding للتأكد من وجود القواعد:
```php
mysqli_query(
    $conn,
    "INSERT IGNORE INTO approval_workflow_rules (...) VALUES (...)"
);
```

### 2. Fallback Rules
في حالة عدم وجود قواعد في قاعدة البيانات، يتم استخدام fallback_map في:
`includes/approval_workflow.php`

### 3. Transaction Support
جميع العمليات محمية بـ transactions لضمان سلامة البيانات
