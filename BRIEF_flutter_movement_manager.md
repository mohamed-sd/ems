# بريف تصميم تطبيق Flutter — مدير الحركة والتشغيل (نظام EMS)

> وثيقة شرح وظيفي لإرسالها إلى Claude Design لإنشاء تصميم تطبيق جوّال (Flutter) يغطّي مهام **مدير الحركة والتشغيل** في نظام إدارة معدات التعدين EMS. أرفِق معها دليل الهوية البصرية إن رغبت.

---

## 1. نظرة عامة

**النظام:** EMS منصّة ويب لإدارة شركات التعدين والمعدات الثقيلة في السودان (عربي، RTL، متعدّد الشركات). نريد بناء **تطبيق جوّال (Flutter)** مخصّص لدور **«مدير الحركة والتشغيل»** يؤدّي مهامه ميدانياً من الهاتف.

**من هو مدير الحركة والتشغيل؟** المسؤول الميداني عن تشغيل المعدات (الآليات الثقيلة) داخل **مشروع تعدين واحد**: يقرّر أي آلية تعمل، على أي عقد ومورّد، في أي وردية (نهار/ليل)، ومَن السائق/المشغّل عليها، ويتابع ساعات العمل والحالة لحظياً، ويُنهي الخدمة أو يُحدّث التعيينات.

**نطاق التطبيق (مهم):** التطبيق مركّز على **شاشتين تشغيليتين فقط** هما كل عمل المدير اليومي:
1. **اللوحة الحيّة / الخريطة** — متابعة لحظية (مكافئ `movement/map_page.php`).
2. **الحركة والتشغيل الموحّدة** — التحكّم والتعديل (مكافئ `movement/movement_operations.php`).
بالإضافة إلى **شاشة الدخول**.

---

## 2. المشروع يُعرف تلقائياً من الدخول (لا توجد شاشة اختيار مشروع)

> هذا تعديل جوهري: **لا تصمّم شاشة لاختيار المشروع.**

حساب مدير الحركة والتشغيل مرتبط مسبقاً بمشروع محدّد (لكل مستخدم `project_id` مخزّن في حسابه). لذلك:
- عند **تسجيل الدخول** يعرف النظام تلقائياً المشروع الذي يعمل عليه المدير.
- التطبيق يدخل مباشرة على مشروعه دون أي اختيار.
- **اسم المشروع وكوده وموقعه** يظهران في رأس الشاشتين كسياق ثابت.
- المدير **مقيّد بمشروعه فقط** ولا يرى/يبدّل مشاريع أخرى.

## 3. المفاهيم والمصطلحات الأساسية (Glossary)

افهم هذه المفاهيم لأنها محور الشاشتين:

- **المشروع (Project):** سياق العمل الثابت (يأتي من حساب المدير عند الدخول).
- **عملية التشغيل (Operation):** سجل يربط **معدة + عقد + مورّد + فئة + نوع وردية + فترة (بداية/نهاية) + ساعات + حالة**. الوحدة الأساسية للعمل.
- **المعدة/الآلية (Equipment):** آلة ثقيلة لها كود واسم ونوع وحالة إتاحة وإحداثيات.
- **المورّد (Supplier) والعقد (Contract):** المعدة تعمل ضمن عقد لمورّد في المشروع.
- **فئة المعدة (Category):** قيمتان: **«أساسي»** أو **«احتياطي»**.
- **نوع الوردية (Shift Type):** ثلاث قيم ثابتة محورية:
  - **D** = نهاري فقط ☀️ · **N** = ليلي فقط 🌙 · **B** = نهاري + ليلي 🔄 (الافتراضي).
  - مهم: العنصر بوردية **B يظهر في قسمي النهار والليل معاً**.
- **السائق/المشغّل (Driver):** يُسند لمعدة عبر سجل **equipment_drivers** بفترة (بداية/نهاية) ونوع وردية وحالة.
- **الحالة (Status):** `1` = ساري (نشط) · `0` = منتهي.
- **النهاية المفتوحة:** تاريخ `2099-12-31` يعني «مستمر بلا نهاية» — **يُعرض للمستخدم كحقل فارغ/«مستمر»** لا كتاريخ.
- **عزل الشركة (Multi-tenant):** المدير يرى بيانات شركته/مشروعه فقط (تلقائي).
- **الصلاحيات:** `can_view` للعرض و`can_edit` لكل عمليات الكتابة — تتحكّم في إظهار الأزرار وقابلية التحرير.

### قواعد عمل صارمة (يجب أن تعكسها الواجهة)
1. **المعدة الواحدة لا تكون في عمليتين ساريتين معاً** — منع + رسالة.
2. **السائق الواحد لا يكون نشطاً على أكثر من تعيين معاً** — عند إضافته لمعدة تُنهى تعييناته السارية السابقة تلقائياً.
3. **التحرير متاح فقط للسجلات السارية** (status=1) ومع صلاحية التعديل؛ المنتهية للعرض فقط.
4. تاريخ النهاية لا يسبق البداية. صيغة التواريخ `YYYY-MM-DD`.

## 4. الشاشة الأولى: اللوحة الحيّة / الخريطة (map_page)

**الغرض:** نظرة واحدة لحظية على الموقع — مَن يعمل، ما المتوقف، ومَن المشغّل، وكم ساعة اشتغلت. هذه شاشة **متابعة (عرض)** أساساً، وهي الواجهة الرئيسية للتطبيق.

**ما تعرضه:**
- **رأس:** اسم المشروع/الكود/الموقع + شارة **«عرض مباشر»** (إحساس لحظي/نبض).
- **بطاقات إحصائية (٥):** إجمالي الموردين · إجمالي الآليات · آليات عاملة · آليات متوقفة · إجمالي المشغّلين.
- **مفتاح ألوان:** عاملة (أخضر) / متوقفة (أحمر) / مشغّل أساسي / احتياطي.
- **تجميع حسب المورّد:** لكل مورّد بطاقة فيها شاراته (إجمالي/عاملة/متوقفة/مشغّلون)، ثم شبكة **بطاقات الآليات** مقسّمة إلى **«آليات عاملة»** و**«آليات متوقفة»**.
- **بطاقة الآلية:** نقطة حالة (أخضر/أحمر) + الكود + النوع + صور المشغّلين الدائرية (أو «لا مشغّل») + **إجمالي الساعات** و**ساعات اليوم**.
- **تفاصيل عند الضغط/المرور:**
  - تفاصيل الآلية: الكود، الاسم، النوع، الماركة/الموديل، المورّد، وللمتوقفة أيضاً: حالة الإتاحة، رقم الهيكل، السيريال، عدد المشغّلين.
  - تفاصيل المشغّل: الاسم، الكود، الهاتف، الكفاءة، الخبرة (وللمتوقفة: نوع الرخصة، خبرة المجال، خبرة الآلية).
- **خريطة (Leaflet/خرائط):** دبابيس للآليات السارية فقط (موقع الآلية، وإلا موقع المشروع)، ونافذة معلومات لكل دبوس: اسم المعدة، النوع، عدد السائقين، نوع الوردية.
- **حالات فارغة:** «لا توجد آليات في هذا المشروع» / «لا توجد آليات لهذا المورّد».
- **تحديث:** سحب للتحديث (pull-to-refresh) مع الإحساس اللحظي.

**ملاحظة تنقّل:** من هذه الشاشة ينتقل المدير إلى شاشة «الحركة والتشغيل الموحّدة» للتعديل.

## 5. الشاشة الثانية: الحركة والتشغيل الموحّدة (movement_operations)

**الغرض:** القلب التشغيلي — تدمج في مكان واحد: **تشغيل المعدات + تعيين السائقين + خريطة** لنفس المشروع، مع **حفظ فوري لكل صف** (دون إعادة تحميل).

### العرض
- **رأس:** «الحركة والتشغيل الموحّدة - {اسم المشروع}».
- **جدولان حسب الوردية:** **تشغيلات النهار** (تضم D و B) و**تشغيلات الليل** (تضم N و B).
- **صف المعدة (رئيسي):** الآلية (كود - اسم)، النوع، الفئة، الوردية، البداية، النهاية، عدّاد السائقين النشطين، شارة الحالة (ساري/منتهي)، أزرار الإجراء.
- **صف فرعي قابل للتوسيع تحت كل معدة:** قائمة سائقي المعدة (السائق، الهاتف، الوردية، بداية/نهاية التعيين، الحالة، إجراء) + نموذج مدمج لإضافة سائق.
- **خريطة:** دبابيس الآليات السارية مع نافذة معلومات (اسم/نوع/عدد سائقين/وردية).
- الترتيب: الساري أولاً ثم الأحدث.

### العمليات (كلها تتطلب صلاحية التعديل، وتُحفظ فوراً لكل صف)
1. **حفظ/تعديل تشغيل معدة قائمة:** تعديل الفئة (أساسي/احتياطي)، نوع الوردية (D/N/B)، الحالة (ساري/منتهي)، تاريخ البداية، تاريخ النهاية. (زر «إنهاء» = ضبط الحالة إلى منتهٍ.) تحقّق: صيغة التواريخ والنهاية بعد البداية.
2. **إضافة تشغيل معدة جديد:** المعدة + العقد + المورّد + نوع المعدة + الفئة + الوردية + البداية + النهاية + إجمالي ساعات الآلية + ساعات الوردية + الحالة. **منع** إضافة معدة لها تشغيل ساري آخر.
3. **حفظ/تعديل تعيين سائق قائم:** نوع الوردية، بداية التعيين، نهاية التعيين (فارغ=مستمر/2099-12-31)، الحالة. (زر «إنهاء» = منتهٍ.)
4. **إضافة سائق لمعدة:** اختيار السائق (تُستثنى من القائمة من لديه تعيين ساري حالياً) + نوع الوردية + بداية (افتراضي اليوم) + نهاية (اختياري). **تُنهى تلقائياً** تعيينات هذا السائق السارية السابقة (سائق واحد لا يكون نشطاً على أكثر من معدة).

### حقول الإدخال (تفصيل)
- **تشغيل:** `equipment_category` (أساسي/احتياطي) · `shift_type` (D/N/B) · `start` (date) · `end` (date) · `status` (1/0). وعند الإضافة أيضاً: المعدة، العقد، المورّد، نوع المعدة، `total_equipment_hours` (رقم عشري)، `shift_hours` (رقم عشري).
- **سائق:** `shift_type` (D/N/B) · `start_date` (date) · `end_date` (date، فارغ=مستمر) · `status` (1/0) · عند الإضافة `driver_id`.

### مصادر القوائم (للتصميم)
- السائقون المتاحون للإضافة: نشطون، ضمن الشركة، مرتبطون بالمشروع أو بلا مشروع، ومُستثنى منهم أصحاب التعيين الساري.
- المعدات/الموردون/العقود: المرتبطة بالمشروع المحدّد (العقود النشطة مرتبة بالأحدث).

## 6. نموذج البيانات (مرجع للمصمم)

- **operations:** id, equipment, equipment_type, equipment_category(أساسي/احتياطي), project_id, contract_id, supplier_id, start, end, days, total_equipment_hours, shift_hours, shift_type(D/N/B), status(1/0).
- **equipment_drivers:** id, equipment_id, driver_id, start_date, end_date(2099-12-31=مستمر), status(1/0), shift_type(D/N/B).
- **equipments:** id, code, name, type, manufacturer, model, serial_number, chassis_number, availability_status, الإحداثيات.
- **drivers:** id, name, driver_code, phone, skill_level, license_type, years_in_field, years_on_equipment, driver_status.
- **suppliers / contracts / project:** المورّد، العقد، المشروع (الاسم/الكود/الموقع/الإحداثيات).
- **timesheet:** مصدر إجمالي ساعات العمل وساعات اليوم لكل معدة.

> ملاحظة: التطبيق سيحتاج طبقة API (REST) تعكس هذه العمليات. الشاشات الحالية تعمل بـ POST مع `action` وردود JSON `{success, message}` وتوكن CSRF — صمّم التدفّقات بافتراض وجود API مكافئ.

## 7. متطلبات غير وظيفية وتوجيهات التصميم

- **عربي بالكامل، اتجاه RTL** في كل الشاشات (المحاذاة، الأيقونات، التواريخ).
- **موجّه للميدان:** أزرار كبيرة، لمس مريح، قراءة تحت الشمس، أقل خطوات لإتمام مهمة.
- **حالة لونية دلالية متّسقة:** أخضر=عامل/ساري، أحمر=متوقف، رمادي=منتهٍ، كهرماني/برتقالي=تمييز الهوية، وأيقونات الوردية ☀️/🌙/🔄.
- **إحساس لحظي:** شارة «عرض مباشر»، سحب للتحديث.
- **حالات فارغة وتحميل وأخطاء** أنيقة.
- **الصلاحيات:** أخفِ/أظهر أزرار التحرير حسب `can_edit`؛ الصفوف المنتهية للعرض فقط.
- **التواريخ:** صيغة موحّدة، والنهاية المفتوحة تُعرض «مستمر».

## 8. المطلوب منك (المخرجات)

1. **تدفّقات المستخدم (User Flows)** للشاشتين + الدخول (المشروع يُعرف تلقائياً، بلا اختيار).
2. **تصميم بصري (Mockups)** لـ:
   - شاشة الدخول (مع توضيح أن المشروع يُحمّل تلقائياً بعد الدخول).
   - اللوحة الحيّة/الخريطة (البطاقات الإحصائية + تجميع الموردين + بطاقات الآليات عاملة/متوقفة + تفاصيل الآلية والمشغّل + الخريطة).
   - الحركة والتشغيل الموحّدة (جدولا النهار/الليل + الصف الفرعي للسائقين + نماذج التعديل والإضافة + الحفظ الفوري + الخريطة).
3. **نظام تصميم (Design System)** متوافق مع هوية النظام (إن أرفقت الدليل فالتزم به): ألوان، خطوط عربية، مكوّنات (بطاقة آلية، بطاقة مورّد، شارات الوردية/الحالة، صفوف قابلة للتوسّع، Bottom sheets، قوائم متسلسلة).
4. شرح موجز لأهم القرارات، وكيف تُعالَج القواعد الصارمة في الواجهة (منع التشغيل المزدوج، الإنهاء التلقائي لتعيينات السائق، التحرير للسارية فقط).

**ابدأ بطرح أي أسئلة توضيحية ضرورية، ثم قدّم تدفّقات المستخدم، ثم التصاميم البصرية للشاشتين.**
